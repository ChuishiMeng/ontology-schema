### 1.2 Baseline 详细配置

#### B1: Direct Prompting (Zero-shot)

**方法**: 直接将问题 + Schema 输入 LLM，生成 SQL

**Prompt 模板**:
```
You are a SQL expert. Given a question and database schema, generate the correct SQL query.

Database Schema:
{schema_ddl}

Question: {question}

Generate the SQL query:
```

**特点**:
- 无 Schema Linking
- 无知识增强
- 最简单基准

**预期性能**:
- Spider: ~60% EX
- BIRD: ~40% EX

---

#### B2: RESDSQL (Decoupled)

**方法**: 解耦 Schema Linking 和 SQL 生成

**架构**:
```
Step 1: Schema Linking
  → 使用分类器选择相关 Schema

Step 2: SQL Generation  
  → 使用选中 Schema 生成 SQL
```

**特点**:
- ✅ 解耦架构
- ✅ 减少 Token 消耗
- ❌ 无 Ontology 层
- ❌ 分类器可能遗漏

**预期性能**:
- Spider: ~75% EX
- BIRD: ~50% EX

---

#### B3: RSL-SQL (Bidirectional)

**方法**: 双向 Schema Linking（Forward + Backward）

**架构**:
```
Forward Linking:
  问题关键词 → Schema 项

Backward Linking:
  Schema 项 → 问题验证

Risk-aware Selection:
  根据风险评分选择 Schema
```

**特点**:
- ✅ 双向验证
- ✅ Risk-aware
- ❌ 无 Ontology 层
- ❌ LLM 验证成本高

**预期性能**:
- Spider: ~82% EX
- BIRD: ~55% EX

---

#### B4: SchemaGraphSQL (Graph-based)

**方法**: 图算法 Schema Linking

**架构**:
```
1. 构建 Schema Graph (表-列-外键)
2. 问题编码为节点
3. 最短路径游走
4. 选择路径上的 Schema
```

**特点**:
- ✅ 图算法高效
- ✅ Zero-shot
- ❌ 无 Ontology 层
- ❌ 仅考虑连接性

**预期性能**:
- Spider: ~78% EX
- BIRD: ~52% EX

---

#### B5: LinkAlign (Multi-Database)

**方法**: 多数据库 Schema Linking + 多智能体辩论

**架构**:
```
Phase 1: Multi-Database Linking
  → 识别问题涉及的数据库

Phase 2: Intra-Database Linking
  → 每个数据库内 Schema Linking

Phase 3: Multi-Agent Debate
  → 多 Agent 辩论验证
```

**特点**:
- ✅ 多数据库支持
- ✅ 多智能体验证
- ❌ 成本高（10+ LLM 调用）
- ❌ 无 Ontology 层

**预期性能**:
- Spider: ~85% EX
- BIRD: ~58% EX

## 🧪 T6.2 消融实验设计

### 2.1 消融实验目的

验证各模块的贡献，识别核心创新点。

### 2.2 消融实验设计

| 实验组 | 配置 | 移除模块 | 验证目标 |
|--------|------|----------|----------|
| **Full System** | 完整系统 | 无 | 基准性能 |
| **w/o Ontology** | 移除 L1 Ontology | Ontology 层 | Ontology 贡献 |
| **w/o Meta-path** | 移除元路径约束 | 元路径推理 | 元路径贡献 |
| **w/o Joint** | 分离召回 | 联合召回算法 | 联合召回贡献 |
| **w/o Minimal** | 不做最小化 | 最小完备集提取 | 最小化贡献 |

### 2.3 消融实验详细设计

#### Exp-A1: w/o Ontology (移除 L1 Ontology 层)

**配置**:
```
移除: L1 Ontology 层
保留: L2 Schema 层 + 联合召回算法
```

**预期效果**:
- 业务术语无法正确映射
- Schema Recall 下降
- EX 下降 15-20%

**验证指标**:
- Term Coverage: 下降至 0%
- Ontology Recall: 下降至 0%
- EX: 下降 15-20%

**示例**:
```
问题: "查询金牌会员的消费总额"
Full System:
  Ontology: 金牌会员 → Member.level='gold'
  Schema: members.level, orders.total_amount
  SQL: 正确

w/o Ontology:
  Schema: 可能错误召回 members (无业务语义)
  SQL: 可能错误
```

---

#### Exp-A2: w/o Meta-path (移除元路径约束)

**配置**:
```
移除: 元路径约束推理
保留: 三层图谱 + 联合召回
召回方式: 简单语义相似度
```

**预期效果**:
- 可能召回无关 Schema
- Schema Precision 下降
- EX 下降 10-15%

**验证指标**:
- Schema Precision: 下降 10%
- Schema Recall: 保持
- EX: 下降 10-15%

**示例**:
```
问题: "查询会员的订单数"
Full System:
  元路径: Member → Order
  Schema: members, orders

w/o Meta-path:
  可能召回: members, orders, products (多余)
  Schema Precision 下降
```

---

#### Exp-A3: w/o Joint (分离召回)

**配置**:
```
移除: 联合召回算法
方式: 先召回 Schema，再召回 Ontology
```

**预期效果**:
- Schema 和 Ontology 信息割裂
- Recall 下降
- EX 下降 5-10%

**验证指标**:
- Ontology Recall: 下降
- Schema Recall: 下降
- EX: 下降 5-10%

---

#### Exp-A4: w/o Minimal (不做最小化)

**配置**:
```
移除: 最小完备集提取
方式: 召回所有相关 Schema（不剪枝）
```

**预期效果**:
- Token 消耗增加
- 噪声 Schema 干扰 SQL 生成
- EX 可能下降或持平

**验证指标**:
- Token Count: 增加 50%
- Schema Precision: 下降
- EX: 可能持平

---

### 2.4 消融实验结果预测（2026-04-15 更新）

| 实验组 | Schema Recall | Ontology Recall | BIRD EX | Token |
|--------|---------------|-----------------|---------|-------|
| **Full System** | 85% | 90% | >65% | ~3K |
| **w/o Ontology** | 60% | 0% | ~55% | ~2K |
| **w/o Meta-path** | 85% | 90% | ~60% | ~4K |
| **w/o Joint** | 70% | 75% | ~58% | ~3K |
| **w/o Minimal** | 85% | 90% | ~62% | ~6K |

**关键发现预测**：
- Ontology 层贡献最大（EX 下降 ~10%，接近 DAIL-SQL 57.41% 水平）
- 元路径约束贡献次之（EX 下降 ~5%）
- 联合召回贡献适中（EX 下降 ~7%）
- 最小完备集影响Token效率（增加 ~100%）

### 3.3 案例研究详细设计

#### Case C1: 语义鸿沟场景

**问题描述**:
```
用户问题: "查询所有金牌会员的消费总额"
业务术语: 金牌会员、消费
Schema术语: members.level='gold', orders.total_amount
```

**对比分析**:
```
Baseline (无 Ontology):
  直接匹配: 无法识别"金牌会员"
  可能召回: members (但不知道筛选条件)
  SQL: SELECT SUM(total_amount) FROM orders  ← 错误，缺少筛选

Ours (有 Ontology):
  Ontology: 金牌会员 → Member.level='gold'
  Schema: members.level, orders.total_amount
  JOIN: members ↔ orders
  SQL: SELECT SUM(o.total_amount) FROM members m JOIN orders o WHERE m.level='gold'
```

**验证目标**:
- Ontology 层解决语义鸿沟
- 业务术语正确映射

---

#### Case C2: 多表关联场景

**问题描述**:
```
用户问题: "查询购买了电子产品的金牌会员数量"
涉及表: members, orders, products
关联: members ↔ orders ↔ products
```

**对比分析**:
```
Baseline (图算法):
  最短路径: members → orders
  遗漏: products 表
  SQL: 缺少产品筛选条件

Ours (元路径约束):
  元路径: Member → Purchase → Product
  Schema: members, orders, products
  约束: 必须包含所有三个表
  SQL: SELECT COUNT(DISTINCT m.id) FROM members m JOIN orders o JOIN products p WHERE m.level='gold' AND p.category='electronics'
```

**验证目标**:
- 元路径约束保证多表关联完整性

---

#### Case C3: 业务规则场景

**问题描述**:
```
用户问题: "查询近期大额订单的平均金额"
业务术语: 近期订单 (7天内), 大额订单 (>10000)
```

**对比分析**:
```
Baseline:
  无法识别业务术语
  SQL: SELECT AVG(total_amount) FROM orders  ← 缺少条件

Ours:
  Ontology: 
    近期订单 → order_date >= DATE_SUB(NOW(), INTERVAL 7 DAY)
    大额订单 → total_amount > 10000
  SQL: SELECT AVG(total_amount) FROM orders WHERE order_date >= ... AND total_amount > 10000
```

**验证目标**:
- Ontology 术语展开支持业务规则

---

#### Case C4: 失败案例分析

**潜在失败场景**:
```
1. Ontology 未覆盖的术语
   问题: "查询 VIP 客户的活跃度"
   问题: Ontology 中无 "VIP 客户" 定义
   
2. 元路径推理失败
   问题: 涉及自连接或复杂子查询
   
3. Schema 召回遗漏
   问题: 问题暗示但未明确提及的表
```

**分析维度**:
- 失败原因
- 与 Baseline 对比
- 改进方向

#### 2. 新增实验设计

##### E4: Ontology 构建质量评估（新增）

**目的**：验证 LLM 半自动构建的 Ontology 质量

**方法**：
1. 在 BIRD 上构建 Ontology（LLM 半自动）
2. 人工评估：覆盖率、准确率、构建时间
3. 对比：vs 手工构建的假设成本

**指标**：
- Ontology Coverage: 覆盖 ground truth SQL 涉及的表/列比例
- Mapping Accuracy: 映射关系正确率
- Construction Time: 实际构建时间

##### E5: Ontology 不完整时的鲁棒性（新增）

**目的**：验证 Ontology 不完整时的降级策略有效性

**方法**：
1. 模拟 Ontology 不完整（随机删除 10%/30%/50% 术语）
2. 运行联合召回 + 降级策略
3. 对比 EX 准确率下降幅度

**预期结果**：
- 10% 删除：EX 下降 < 5%
- 30% 删除：EX 下降 < 15%
- 50% 删除：仍优于纯 Schema Linking

##### E6: 与 ATHENA/Sequeda 的对比（新增）

**目的**：与 OBDA 先驱工作对比

**方法**：
- 无法直接复现 ATHENA（自定义数据集）
- 在 Sequeda Benchmark（公开）上评估本方法
- 对比：SQL 准确率 vs KG 准确率 vs 本研究

**预期结果**：
- 本研究应接近或优于 Sequeda 的 KG 准确率 (54.2%)
- 且构建成本显著低于 Sequeda

#### 3. 成功标准修订（基于2026-04-15文献解读）

| 指标 | 原目标（不合理） | 修订目标 | 实证参考 |
|------|-----------------|----------|----------|
| **BIRD EX** | >85% ❌ | **>65%** | CHESS 71.10%, CodeS 63.62%, DAIL-SQL 57.41% |
| **vs Baseline 提升** | >10% ❌ | **>5%** | vs DAIL-SQL 57.41% → 目标 >62% |
| **Ontology 贡献** | >15% ⚠️ | **>5-10%** | w/o Ontology 应接近纯Schema方法 |
| **LLM 调用** | ≤2次 ⚠️ | **≤4次** | CHESS ~6次, DIN-SQL 4+次 |
| **构建成本** | ≤2小时 | **≤2小时/DB** | ATHENA 数周, Sequeda 数周 |

**实证支撑**（来自文献解读）：
1. **语义鸿沟真实存在**：EHRSQL医护"体温"→Schema`temperature c (calc)`，T5+Schema≈T5 alone
2. **Schema理解瓶颈**：DAIL-SQL Spider86.6%→BIRD57.41%，下降30个百分点
3. **企业级性能灾难**：Spider 2.0 o1仅21.3%，Schema Linking错误27.6%
4. **Ontology必要性**：ATHENA 10年前验证，Sequeda KG提升3x准确率

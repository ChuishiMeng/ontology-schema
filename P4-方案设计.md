### 与 LLM-based SOTA 的架构对比（v2.1 新增）

```
CHESS (2024, BIRD SOTA 71.10%):
  NL → IR(检索) → SS(Schema剪枝) → CG(SQL生成) → UT(修订)
  特点：四Agent渐进式剪枝(~10列)，表面级NL匹配，~6次LLM调用

CodeS (2024, 开源SOTA 63.62%):
  NL → Schema Filter(BM25/LCS) → SQL Generation
  特点：字面+注释理解，1次LLM调用，缺深层语义

DAIL-SQL (2024, Prompt天花板 57.41%):
  NL → Few-shot Prompt → SQL
  特点：纯prompt engineering，1次LLM调用，复杂Schema性能骤降30%

本研究 (2026, 目标>65%):
  NL → [Ontology语义理解] → 联合召回(最小完备集) → SQL
  特点：Ontology深层语义，最小完备集(~5列)，元路径约束，2-4次LLM调用
```

### 与 ATHENA 的架构对比

```
ATHENA (2016):
  NL → [TI索引 + 同义词] → OQL → [手工Mapping] → SQL
  特点：全量查询、手工Ontology、传统NLP

本研究 (2026):
  NL → [LLM理解] → 联合召回(最小完备集) → SQL
  特点：精准召回、LLM半自动Ontology、元路径约束
```

## 🏗️ 模块一：Ontology 半自动构建（新增）

### 1.1 构建流程

```
输入: 数据库 Schema DDL + 业务文档(可选)
输出: Ontology O (概念 + 属性 + 关系 + 术语映射)

Step 1: Schema 分析
  - LLM 解析 DDL，提取表、列、外键关系
  - 生成初始概念-属性-关系草案

Step 2: 业务文档增强（如果有）
  - LLM 从文档提取业务术语和计算规则
  - 映射业务术语到 Schema 字段

Step 3: Ontology 生成
  - 合并 Step 1 + Step 2
  - 输出: 概念(表级) + 属性(列级) + 关系(外键) + 术语(值级)

Step 4: 人工校验
  - 领域专家审核 Ontology 质量
  - 修正错误映射
  - 补充遗漏的业务术语

Step 5: 质量评估
  - 覆盖率: Ontology 覆盖了多少 ground truth SQL 涉及的表/列
  - 准确率: 映射关系正确的比例
```

### 1.2 Ontology 结构（两层，非三层）

```
L1: Ontology 层（业务知识）
    ├── 概念: Member, Product, Order
    ├── 属性: Member.level, Order.total_amount
    ├── 关系: Member → Purchase → Product
    └── 术语: 金牌会员 = level='gold', 高价值 = spend>1000

L2: Schema 层（物理数据库）
    ├── 表: members, products, orders
    ├── 列: members.level, orders.total_amount
    └── 外键: orders.member_id → members.id

L1→L2 映射:
    ├── 直接映射 (1:1): Member → members
    ├── 一对多映射 (1:N): 客户 → users + user_profiles
    └── 计算型映射: 客单价 = SUM(amount)/COUNT(DISTINCT user_id)
```

### 1.3 LLM 调用分析

| 步骤 | LLM 调用 | 说明 |
|------|---------|------|
| Step 1 Schema 分析 | 1 次 | 解析 DDL |
| Step 2 文档增强 | 1-2 次 | 提取术语 |
| Step 3 合并 | 0 次 | 规则合并 |
| **构建总计** | **2-3 次/数据库** | 一次性成本 |

vs ATHENA: 领域专家数周 → LLM 2-3 次调用 + 人工校验 1-2 小时

## 🏗️ 模块二：联合召回算法（细化版）

### 2.1 算法流程

```
Algorithm: Ontology-Schema 联合召回

Input: 
  - 问题 q
  - Ontology O (L1)
  - Schema S (L2)
  - L1→L2 映射 M
  
Output:
  - 最小完备集 MCS = {tables*, columns*, knowledge*}

Step 1: 问题理解 [LLM 调用: 1次]
  1.1 LLM 提取问题中的实体、属性、条件
  1.2 输出结构化: {entities, attributes, conditions, intent}
  1.3 Fallback: 如果 LLM 提取失败，用关键词匹配 + 向量检索

Step 2: L1 Ontology 召回 [无 LLM 调用]
  2.1 实体匹配: 问题实体 → Ontology 概念 (向量相似度)
  2.2 关系发现: 通过 Ontology 关系图发现关联概念
  2.3 术语展开: 匹配业务术语 → 展开为具体条件
  2.4 输出: 候选 Ontology 子图 G_L1

Step 3: L2 Schema 映射 [无 LLM 调用]
  3.1 概念→表: G_L1 中的概念通过映射 M 转为表
  3.2 属性→列: Ontology 属性映射为数据库列
  3.3 关系→JOIN: Ontology 关系映射为外键 JOIN 路径
  3.4 处理复杂映射:
      - 1:N 映射: 展开所有相关表
      - 计算型映射: 保留公式作为 knowledge*
  3.5 输出: 候选 Schema 子图 G_L2

Step 4: 元路径约束验证 [可选 LLM 调用: 0-1次]
  4.1 从 G_L2 构建候选元路径集合
  4.2 规则验证 (无需 LLM):
      - 路径覆盖所有问题实体
      - 路径中无悬空节点（每个节点都有外键连接）
      - 路径长度 ≤ max_length (默认 4)
  4.3 如果多条路径残留 (可选):
      - LLM 判断哪条更符合问题语义
  4.4 路径排序: 语义相关性 × 1/路径长度
  4.5 输出: 约束后的 Schema 子图 G_constrained

Step 5: 最小完备集提取 [无 LLM 调用]
  5.1 贪心算法:
      - 初始化 MCS = 所有必需表（Step 4 输出）
      - 添加 JOIN 连接表（保证表间可达）
      - 逐个检查列：是否出现在问题/条件/JOIN 中
      - 移除冗余项（移除后仍完备的最大子集）
  5.2 完备性检查:
      - 确保所有问题提及的概念有对应表
      - 确保 JOIN 路径连通
  5.3 输出: MCS = {tables*, columns*, knowledge*}

return MCS
```

### 2.2 LLM 调用次数分析

| 场景 | Step 1 | Step 4 | SQL 生成 | 总计 |
|------|--------|--------|---------|------|
| **Best case** | 1 | 0 | 1 | **2 次** |
| **Typical** | 1 | 1 | 1 | **3 次** |
| **Worst case** | 1 (retry) | 1 | 1 | **4 次** |

> 修正：原声称"1-2次"不准确。实际 typical 3 次，best 2 次，worst 4 次。
> 对比：RSL-SQL 5+次，LinkAlign 10+次，SchemaGraphSQL 1 次。

### 2.3 Ontology 不完整时的降级策略

| 场景 | 策略 |
|------|------|
| 术语不在 Ontology 中 | 向量检索 fallback → 直接在 Schema 中搜索 |
| 映射缺失 | LLM 辅助推断映射 → 人工确认后补充 |
| 关系缺失 | 基于外键关系自动补全 |
| Ontology 完全不可用 | 退化为纯 Schema Linking（RSL-SQL 等价） |

## 🧭 模块三：元路径约束推理（修订版）

### 3.1 元路径定义（不变）

$$P = (E_1, R_1, E_2, ..., E_n)$$

### 3.2 约束规则（修订）

| 规则 | v1.0 | v2.0 修订 |
|------|------|----------|
| 最短路径优先 | ✅ | ❌ 改为 **语义相关性 × 1/路径长度** 加权排序 |
| 语义一致性 | ✅ | ✅ 保留 |
| 完备性保证 | ✅ | ✅ 保留 |

**修订原因**：审阅指出"最短路径优先"可能选错（如 Department→Sales 应为 Department→Employee→Sales）

### 3.3 排序公式

$$\text{Score}(P) = \alpha \cdot \text{SemanticSim}(P, Q) + (1-\alpha) \cdot \frac{1}{|P|}$$

其中 $\alpha$ 控制语义相关性和路径长度的权衡。

## 📦 模块四：最小完备集提取（形式化版）

### 4.1 形式化定义

**完备性**：
$$\text{Complete}(M, Q) \iff \text{Tables}(M) \supseteq \text{RequiredTables}(Q) \land \text{Columns}(M) \supseteq \text{RequiredColumns}(Q)$$

**最小性**：
$$M^* = \arg\min_{M} |M| \quad \text{s.t.} \quad \text{Complete}(M, Q)$$

### 4.2 计算复杂度

等价于 **Weighted Set Cover** 问题，NP-hard。

### 4.3 近似算法

使用贪心算法，近似比 $H_n = \sum_{i=1}^{n} \frac{1}{i} = O(\log n)$。

```
贪心算法:
1. 初始化 M = ∅
2. 重复: 选择使完备性提升最大的元素加入 M
3. 直到 M 完备
4. 反向检查: 从 M 中移除冗余元素（移除后仍完备）
```

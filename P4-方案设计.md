## 🏗️ 模块一：三层图谱结构 (HKSG)

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **统一表示** | 业务知识与物理 Schema 统一表示 |
| **层次分离** | 业务层与物理层解耦，便于维护 |
| **约束传递** | 元模型约束 Ontology 和 Schema |
| **语义桥接** | Ontology 桥接业务术语与 Schema 术语 |

### 1.2 层次结构

#### L0: 元模型层 (Meta-Model Layer)

**作用**：定义 HKSG 的结构规范，约束 L1 和 L2。

```
元模型定义:

EntityType:
  - name: String
  - attributes: List[Attribute]
  - constraints: List[Constraint]

RelationType:
  - name: String
  - source: EntityType
  - target: EntityType
  - cardinality: {1:1, 1:N, N:M}

Attribute:
  - name: String
  - type: {String, Int, Float, Date, ...}
  - constraints: List[Constraint]

Constraint:
  - type: {Unique, NotNull, ForeignKey, ...}
  - value: Any
```

**约束规则**：
- L1 的业务实体必须继承 L0 的 EntityType
- L2 的数据库表必须映射到 L1 的业务实体

#### L1: Ontology 层 (Business Knowledge Layer)

**作用**：表示业务领域的实体、关系和术语。

```
业务实体示例:

Entity: Member (会员)
  - attributes:
    - member_id: String (会员ID)
    - level: String (会员等级)
    - register_date: Date (注册日期)
  - business_terms:
    - 金牌会员: level = 'gold'
    - 银牌会员: level = 'silver'
    - 新会员: register_date within 30 days

Entity: Product (商品)
  - attributes:
    - product_id: String
    - name: String
    - category: String
    - price: Float
  - business_terms:
    - 热销商品: sales_count > 1000
    - 新品: create_date within 7 days

Entity: Order (订单)
  - attributes:
    - order_id: String
    - member_id: String
    - total_amount: Float
    - order_date: Date
  - business_terms:
    - 大额订单: total_amount > 10000
    - 近期订单: order_date within 7 days
```

```
业务关系示例:

Relation: Purchase (购买)
  - source: Member
  - target: Product
  - via: Order
  - cardinality: N:M

Relation: Belong (属于)
  - source: Order
  - target: Member
  - cardinality: N:1
```

#### L2: Schema 层 (Physical Schema Layer)

**作用**：表示数据库的表、字段和外键关系。

```
Schema 定义:

Table: members
  - columns:
    - id (PK): Integer
    - level: String
    - register_date: Date
  - mapping: Member (L1)

Table: products
  - columns:
    - id (PK): Integer
    - name: String
    - category: String
    - price: Float
  - mapping: Product (L1)

Table: orders
  - columns:
    - id (PK): Integer
    - member_id (FK): Integer → members.id
    - total_amount: Float
    - order_date: Date
  - mapping: Order (L1)
  - foreign_keys:
    - member_id → members.id
```

### 1.3 层间映射

**L1 → L2 映射**：

```
Ontology 实体 → Schema 表:

Member → members
  - member_id → id
  - level → level
  - register_date → register_date

Product → products
  - product_id → id
  - name → name
  - category → category
  - price → price

Order → orders
  - order_id → id
  - member_id → member_id
  - total_amount → total_amount
  - order_date → order_date
```

**业务术语 → SQL 条件映射**：

```
金牌会员 → members.level = 'gold'
银牌会员 → members.level = 'silver'
新会员 → members.register_date >= DATE_SUB(NOW(), INTERVAL 30 DAY)
热销商品 → products.sales_count > 1000
大额订单 → orders.total_amount > 10000
```

## 🔍 模块二：联合召回算法

### 2.1 算法目标

| 目标 | 说明 |
|------|------|
| **完整性** | 召回所有相关的 Schema 项和业务知识 |
| **最小性** | 仅召回必需的项，减少噪声 |
| **正确性** | 召回的项语义正确，满足问题需求 |
| **效率** | 快速召回，减少 LLM 调用 |

### 2.2 算法流程

```
Algorithm: 联合召回算法

Input: 
  - 问题 q
  - 三层图谱 HKSG = (L0, L1, L2)
  
Output:
  - 相关 Schema 项 S* = {tables, columns}
  - 相关业务知识 K* = {entities, relations, terms}

步骤:

Step 1: 问题理解
  1.1 实体识别: 识别问题中的业务实体 E_q
  1.2 意图识别: 识别查询意图 I_q (统计/筛选/聚合)
  1.3 术语映射: 识别业务术语 T_q

Step 2: L1 召回 (Ontology 层)
  2.1 实体召回: 从 E_q 出发，召回相关实体 E*
  2.2 关系召回: 召回实体间的关系 R*
  2.3 术语展开: 展开业务术语到具体条件

Step 3: L2 召回 (Schema 层)
  3.1 表映射: E* 映射到数据库表 T*
  3.2 字段映射: 实体属性映射到字段 C*
  3.3 外键补全: 通过外键关系补全连接表

Step 4: 元路径约束验证
  4.1 构建元路径: 从 T* 构建候选元路径
  4.2 语义验证: 验证元路径是否满足问题语义
  4.3 路径剪枝: 移除语义不相关的路径

Step 5: 最小完备集提取
  5.1 完备性检查: 确保所有必需项已召回
  5.2 最小化剪枝: 移除冗余项
  5.3 输出最终 Schema S* 和知识 K*

return (S*, K*)
```

### 2.3 关键算法详解

#### Step 1: 问题理解

**实体识别 (NER)**：
```
输入: "查询所有金牌会员的消费总额"

步骤:
1. NER 识别: "金牌会员" → EntityCandidate
2. 术语库匹配: "金牌会员" → Member.level = 'gold'
3. 输出: E_q = {Member}
```

**意图识别**：
```
意图类型:
- 统计: COUNT, SUM, AVG
- 筛选: WHERE conditions
- 聚合: GROUP BY
- 排序: ORDER BY

示例: "消费总额" → 意图 = {聚合: SUM, 目标: Order.total_amount}
```

#### Step 4: 元路径约束验证

**元路径定义**：
```
元路径 = 实体类型序列 + 关系类型序列

示例:
Member → Purchase → Product
  = (Member, Product) + (Purchase)
  
Schema 映射:
members → orders → products
  = (members, orders, products) + (orders.member_id, orders.product_id)
```

**语义验证**：
```
问题: "查询每个会员的订单数"
候选元路径: Member → Purchase → Product
语义分析: 问题仅涉及会员和订单，不涉及商品
判定: 元路径包含冗余实体 Product → 剪枝

优化后元路径: Member → Order
Schema 映射: members → orders
```

#### Step 5: 最小完备集提取

**完备性定义**：
```
完备集: 召回的 Schema 项能够生成满足问题需求的 SQL

示例:
问题: "查询金牌会员的消费总额"
必需 Schema:
  - members (筛选金牌会员)
  - orders (计算消费总额)
  - orders.member_id (JOIN 条件)

非必需 Schema:
  - products (问题不涉及商品)
```

**最小化算法**：
```
Algorithm: 最小完备集提取

Input: 召回的 Schema 项 S_candidate
Output: 最小完备集 S_minimal

S_minimal = ∅
for each item in S_candidate:
    if item 是 SQL 必需的:
        S_minimal.add(item)
    else:
        continue  # 剪枝

return S_minimal
```

## 🧭 模块三：元路径约束推理

### 3.1 元路径定义

**形式化定义**：
```
元路径 P = (E_1, R_1, E_2, R_2, ..., E_n)

其中:
- E_i: 实体类型 (来自 L1 Ontology)
- R_i: 关系类型 (来自 L1 Ontology)
- 长度 = n - 1
```

**示例**：
```
P1: Member → Purchase → Product
    长度 = 2
    语义: 会员购买商品

P2: Member → Order
    长度 = 1
    语义: 会员的订单

P3: Member → Order → Product
    长度 = 2
    语义: 会员通过订单购买商品
```

### 3.2 元路径类型

| 类型 | 模式 | 示例 | 适用场景 |
|------|------|------|---------|
| **简单路径** | E_1 → E_2 | Member → Order | 单表或简单 JOIN |
| **多跳路径** | E_1 → E_2 → E_3 | Member → Order → Product | 多表 JOIN |
| **循环路径** | E_1 → E_2 → E_1 | Member → Order → Member | 自连接 |

### 3.3 元路径约束规则

**规则 1: 语义一致性**
```
如果问题提及实体 E_1 和 E_3，则元路径必须包含 E_1 和 E_3

示例:
问题: "查询会员购买的商品"
实体: Member, Product
必需元路径: Member → ... → Product
```

**规则 2: 最短路径优先**
```
如果存在多条元路径，优先选择最短路径

示例:
路径1: Member → Order → Product (长度 2)
路径2: Member → Order → OrderItem → Product (长度 3)

选择: 路径1 (更短)
```

**规则 3: 完备性保证**
```
元路径必须覆盖问题中提及的所有实体

示例:
问题: "查询金牌会员购买的商品数量"
实体: Member, Product
必需覆盖: Member 和 Product

正确路径: Member → Order → Product
错误路径: Member → Order (缺少 Product)
```

### 3.4 元路径推理算法

```
Algorithm: 元路径推理

Input: 
  - 问题中的实体集合 E_q
  - HKSG 三层图谱

Output:
  - 元路径集合 P*

步骤:

Step 1: 确定起点和终点
  - 起点: 问题中明确提及的实体
  - 终点: 问题中明确提及的实体

Step 2: 枚举候选路径
  - 使用 BFS/DFS 枚举起点到终点的所有路径
  - 限制最大路径长度 (如 max_length = 4)

Step 3: 语义验证
  - 检查路径是否覆盖所有 E_q
  - 检查路径语义是否与问题意图一致

Step 4: 路径排序
  - 按路径长度排序 (短路径优先)
  - 按语义相关性排序

Step 5: 选择最优路径
  - 选择 Top-K 条路径作为候选
  - 通过 LLM 验证最终选择

return P*
```

## 📦 模块四：最小完备集提取

### 4.1 问题定义

**目标**：在保证 SQL 可生成的前提下，最小化召回的 Schema 项。

**形式化**：
```
给定问题 q 和候选 Schema 项 S_candidate
找到 S_minimal ⊆ S_candidate，满足:
1. 完备性: S_minimal 可以生成满足 q 的 SQL
2. 最小性: |S_minimal| 最小
```

### 4.2 算法设计

```
Algorithm: 最小完备集提取

Input: 
  - 问题 q
  - 候选 Schema 项 S_candidate = {T_1, ..., T_n, C_1, ..., C_m}
  
Output:
  - 最小完备集 S_minimal

步骤:

Step 1: 初始化
  S_minimal = ∅
  S_required = ∅

Step 2: 识别必需表
  for each 表 T in S_candidate:
      if T 包含问题所需的字段:
          S_required.add(T)

Step 3: JOIN 路径补全
  for each T in S_required:
      if T 与其他必需表无直接连接:
          找到最短 JOIN 路径
          将路径上的表加入 S_minimal

Step 4: 字段筛选
  for each 字段 C in S_candidate:
      if C 出现在问题中 或 C 是 JOIN 条件:
          S_minimal.add(C)

Step 5: 验证完备性
  if S_minimal 可以生成 SQL:
      return S_minimal
  else:
      添加缺失项并重复 Step 3-4

return S_minimal
```

### 4.3 示例

```
问题: "查询所有金牌会员的消费总额"

Step 1: 问题理解
  - 实体: Member (金牌会员)
  - 意图: 聚合 SUM(Order.total_amount)
  - 条件: Member.level = 'gold'

Step 2: 识别必需表
  - members: 包含 level 字段 (筛选条件)
  - orders: 包含 total_amount 字段 (聚合目标)
  - S_required = {members, orders}

Step 3: JOIN 路径补全
  - members.id ← orders.member_id (外键关系)
  - 无需额外表

Step 4: 字段筛选
  - members.level: 筛选条件
  - members.id: JOIN 条件
  - orders.member_id: JOIN 条件
  - orders.total_amount: 聚合目标

Step 5: 最小完备集
  S_minimal = {
    tables: {members, orders},
    columns: {members.id, members.level, orders.member_id, orders.total_amount}
  }
```

## 📊 与现有方法对比

| 维度 | LinkAlign (P01) | RSL-SQL (P04) | SchemaGraphSQL (P02) | **本研究** |
|------|----------------|---------------|---------------------|-----------|
| **知识层** | 无 | 无 | 无 | **三层图谱** |
| **召回方式** | 多智能体辩论 | 双向链接 | 最短路径 | **元路径约束** |
| **约束机制** | LLM 验证 | LLM 验证 | 图算法 | **元路径约束** |
| **LLM 调用** | 10+ 次 | 5+ 次 | 1 次 | **1-2 次** |
| **业务术语** | ❌ 不支持 | ❌ 不支持 | ❌ 不支持 | **✅ 支持** |

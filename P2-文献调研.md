### 核心发现(v2.1 更新)

| 发现 | 说明 | 实证来源 |
|------|------|----------|
| **OBDA 已有 10 年历史** | ATHENA (VLDB 2016) Ontology-driven NL→SQL,Precision 99% | P23 解读 |
| **KG 显著提升准确率** | Sequeda: KG 提升 GPT-4 准确率 3x (16.7%→54.2%) | P16 解读 |
| **Schema 理解是核心瓶颈** | DAIL-SQL Spider86.6%→BIRD57.41%,下降30个百分点 | P22 解读 ⭐ |
| **语义鸿沟真实存在** | EHRSQL: 医护"体温"→Schema`temperature c (calc)`,T5+Schema≈T5 alone | P27 解读 ⭐ |
| **企业级性能灾难** | Spider 2.0: o1仅21.3% vs Spider1.0 91.2%,Schema Linking错误27.6% | P28 解读 ⭐ |
| **BIRD SOTA 渐进剪枝** | CHESS 四Agent架构,BIRD test 71.10%,LLM调用减少83% | P24 解读 ⭐ |
| **开源模型超越GPT-4** | CodeS-3B Spider85.4% > DIN-SQL+GPT-4,缺深层语义建模 | P25 解读 |
| **高 Schema 复杂度时失效** | Sequeda: >5 表时 SQL 准确率 0% → 支持最小完备集 | P16 解读 |

### 关键结论(v2.1 更新)

1. ✅ **Schema理解是核心瓶颈**:DAIL-SQL/CHESS/CodeS/C3 都停留在表面级NL匹配,缺乏深层语义建模
2. ✅ **语义鸿沟实证证据**:EHRSQL 直接展示业务术语→Schema术语的映射失败
3. ✅ **KG/Ontology 有效**:Sequeda KG提升3x,ATHENA 99% Precision
4. ⚠️ **"Ontology 空白"需重新定位**:ATHENA 10年前验证,但**未迁移到LLM时代Schema Linking**
5. ✅ **最小完备集必要**:>5表全量Schema失效,企业级场景灾难性下降
6. ✅ **Baseline完整覆盖**:CHESS(BIRD SOTA)+CodeS(开源)+DAIL-SQL(prompt)+DIN-SQL(分解)+ATHENA(OBDA)

## 🔬 新增论文深度分析(v2.1 补充)

### P24: CHESS (2024) ⭐⭐⭐⭐⭐ BIRD SOTA

**核心贡献**:
- 四Agent架构:IR(检索) → SS(Schema剪枝) → CG(SQL生成) → UT(修订)
- 渐进式剪枝:96列 → filter_column → select_tables → ~10列
- BIRD test EX: **71.10%**,LLM调用减少83%

**与本研究对比**:
| 维度 | CHESS | 本研究 |
|------|--------|--------|
| Schema理解 | 表面级NL匹配 | Ontology深层语义 |
| 剪枝策略 | 渐进式(LLM多次调用) | 最小完备集(1-2次) |
| 知识层 | 无 | Ontology概念层 |

**借鉴价值**: 渐进式剪枝思路、修订循环、模块化设计便于对比

---

### P25: CodeS (2024) ⭐⭐⭐⭐ 开源SOTA

**核心贡献**:
- 21.5GB SQL-centric语料增量预训练
- Schema Filter + BM25/LCS筛选
- CodeS-3B Spider 85.4% > DIN-SQL+GPT-4

**关键洞察**: Schema理解停留在"字面+注释",缺乏深层语义建模 → 本研究切入点

### P26: C3 (2023) ⭐⭐⭐ Zero-shot SOTA

**核心贡献**:
- 三组件:Clear Prompting + Calibration with Hints + Consistent Output
- Spider Zero-shot SOTA 82.3%

**关键洞察**: Schema Linking用LLM直接识别,无Ontology → 本研究切入点

---

### P27: EHRSQL (2024) ⭐⭐⭐⭐ 语义鸿沟实证

**核心贡献**:
- 医护真实需求:222人调研收集1,742条问题
- **语义鸿沟直接证据**: 医护说"体温"→Schema`d_items.label='temperature c (calc)'`
- **关键发现**: T5+Schema ≈ T5 alone(Schema序列化无效)

**启示**: 需要额外业务知识层(Ontology)

### P28: Spider 2.0 (ICLR 2025 Oral) ⭐⭐⭐⭐⭐ 企业级基准

- 632企业级任务,Schema平均812列(vs Spider1.0 27列)
- **性能灾难**: o1-preview 21.3% vs Spider1.0 91.2%
- **错误分析**: Schema Linking占27.6%错误

**启示**: 企业级语义鸿沟的最直接量化证据

### P22: DAIL-SQL (VLDB 2024) ⭐⭐⭐⭐ Prompt天花板

## 🎯 研究空白分析(v2.1 更新)

### G1: OBDA 未迁移到 LLM 时代 Schema Linking(重定义)

**证据**:
- ATHENA (2016) 已做 Ontology-driven NL→SQL,Precision 99%
- Sequeda (2023) KG提升GPT-4 3x准确率
- **但**: ATHENA手工Ontology + 传统NLP,Sequeda仅评估无方法
- **LLM-based SOTA**: CHESS/DAIL-SQL/DIN-SQL/CodeS **未引入Ontology层**

**差异化**: LLM半自动构建 + 最小完备集 + 元路径推理 + BIRD评估

### G2: Schema 理解停留在表面级NL匹配(新证据)

**证据**:
- CHESS: 渐进式剪枝,表面级NL语义匹配
- CodeS: 字面+注释理解,缺深层语义
- DAIL-SQL: Spider86.6%→BIRD57.41%,复杂Schema性能骤降
- C3: LLM直接Schema Linking,无Ontology

### G3: Ontology 构建成本阻碍落地(新增)

**证据**:
- ATHENA: 领域专家数周手工OWL2
- Sequeda: 手工OWL+R2RML
- OPTIQUE: 半自动但仍需大量人工

**本研究方案**: LLM半自动构建,目标≤2小时/DB

### G4: 企业级语义鸿沟量化证据(新增)

**证据**:
- Spider 2.0: o1仅21.3% vs Spider1.0 91.2%
- EHRSQL: 医护"体温"→Schema`temperature c (calc)`
- EHRSQL: T5+Schema ≈ T5 alone(Schema序列化无效)

**启示**: 企业级场景Ontology必要性实证

## 📊 执行摘要

### 核心发现

| 发现 | 说明 |
|------|------|
| **Schema Linking 是关键瓶颈** | 决定 Text-to-SQL 系统的上限 |
| **现有方法缺乏业务知识层** | 无法处理业务术语与 Schema 的语义鸿沟 |
| **图谱方法有效但未被充分利用** | 图算法、GNN 方法已证明有效，但未与 Ontology 结合 |
| **多智能体协作成为趋势** | 但成本高、延迟大，需要更高效的协作机制 |

### 关键结论

1. **召回优先**：Schema Linking 的 Recall 比 Precision 更重要（遗漏比噪声更致命）
2. **双向验证有效**：Forward（问题→Schema）+ Backward（Schema→问题）提升召回
3. **图结构表示优于纯文本**：联合编码问题与 Schema 效果更好
4. **知识增强是空白**：现有方法均未系统引入 Ontology 层

## 🎯 核心问题分析

### 问题 1: Schema Linking 的风险

**P04 RSL-SQL** 系统分析了 Schema Linking 的两大风险：

| 风险 | 说明 | 后果 |
|------|------|------|
| **遗漏必要元素** | 丢失 SQL 生成所需的表/列 | SQL 无法生成或错误 |
| **破坏结构完整性** | 切断外键关系 | JOIN 推理失败 |

**关键指标**：
- Strict Recall: 94%（必须保证高召回）
- Column Reduction: 83%（同时减少噪声）

**解决方案**：
- **双向链接**：Forward（问题→Schema）+ Backward（Schema→问题）
- **二选一策略**：Voting 避免极端情况
- **上下文增强**：添加外键、主键信息

### 问题 2: 多数据库环境

**P01 LinkAlign** 首次系统分析了多数据库场景的挑战：

```
多数据库 Schema Linking 两阶段：

阶段1: Database Retrieval
├── 从大量 Schema 池中选择目标数据库
└── 挑战：语义相似数据库易混淆

阶段2: Schema Item Grounding  
├── 在选中的数据库中定位相关表/列
└── 挑战：大规模 Schema，冗余字段多
```

**解决方案**：
- **Multi-round Semantic Enhanced Retrieval**：Query Rewriting 语义对齐
- **Irrelevant Information Isolation**：多智能体辩论过滤噪声

### 问题 3: 语义鸿沟

**P10 SEDE** 揭示了真实世界 Text-to-SQL 的语义鸿沟问题：

| 学术数据集 | 真实世界 (SEDE) |
|-----------|----------------|
| 问题明确提及列名 | 问题可能不提及列名 |
| 标准化问题风格 | 口语化、多样化 |
| SQL 相对简单 | SQL 复杂（嵌套、窗口函数）|

**根本原因**：用户使用业务术语提问，而 Schema 使用技术术语，存在语义鸿沟。

**现有方法局限**：均未引入 Ontology 层桥接业务术语与 Schema 术语。

## 🔍 研究空白分析

### 空白 1: 缺乏 Ontology 层

| 现有方法 | 局限 |
|---------|------|
| 全部 13 篇论文 | 均未系统引入 Ontology 层 |
| P07 DataFactory | 有 KG 团队，但未专门处理业务术语 |
| P06 GraphMatcher | 处理 Ontology Matching，但未与 Schema 结合 |

**机会**：引入 Ontology 层，桥接业务术语与 Schema 术语。

### 空白 2: Schema Linking 与知识召回割裂

| 现有方法 | 局限 |
|---------|------|
| Schema Linking 方法 | 仅召回 Schema 项 |
| 知识图谱方法 | 仅推理知识，不涉及 SQL |
| P07 DataFactory | 两团队独立协作，未联合召回 |

**机会**：Schema + Ontology 联合召回，一次性获取所有相关知识。

### 空白 3: 缺乏结构约束机制

| 现有方法 | 局限 |
|---------|------|
| P02 SchemaGraphSQL | 用最短路径，但无语义约束 |
| P04 RSL-SQL | 双向验证，但依赖 LLM |
| P09 RESDSQL | Skeleton 约束，但仅约束 SQL 结构 |

**机会**：通过元路径约束，保证召回的语义正确性。

### 空白 4: 多智能体成本高

| 现有方法 | 成本 |
|---------|------|
| P01 LinkAlign | 多轮检索 + 多智能体辩论，延迟高 |
| P05 MAG-SQL | 4 个 Agent，多次 LLM 调用 |
| P07 DataFactory | Database + KG 双团队，协作成本高 |

**机会**：通过图谱结构化推理，减少 LLM 调用次数。

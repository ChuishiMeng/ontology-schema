## 🔬 空白详细分析

### G1: 缺乏 Ontology 层 ⭐⭐⭐⭐⭐

#### 问题定义

**语义鸿沟**：用户使用业务术语提问，Schema 使用技术术语，两者无法直接对齐。

```
示例：
用户问题: "查询所有金牌会员的消费总额"
业务术语: 金牌会员、消费
Schema术语: members.level='gold', orders.total_amount

问题: "金牌会员" ≠ "members.level='gold'"
```

#### 现有方法局限

| 论文 | 方法 | 局限 |
|------|------|------|
| P01-P15 | 全部 13 篇 | 均未引入 Ontology 层 |
| P07 DataFactory | KG 团队协作 | KG 用于推理，非术语对齐 |
| P06 GraphMatcher | Ontology Matching | 对齐两个 Ontology，非 Ontology-Schema |

#### 技术可行性分析

| 技术组件 | 现有基础 | 可借鉴论文 | 可行性 |
|---------|---------|-----------|--------|
| Ontology 构建 | 领域知识图谱构建成熟 | 知识图谱构建方法 | ✅ 高 |
| Ontology-Schema 映射 | Ontology Matching 技术 | P06 GraphMatcher | ✅ 高 |
| 术语对齐 | 语义相似度、词汇匹配 | P04 RSL-SQL 双向链接 | ✅ 高 |

#### 解决方案思路

**三层图谱结构**：

```
L0: 元模型层 (Meta-Model Layer)
    └── 约束 L1、L2 的结构规范

L1: Ontology 层 (Business Knowledge Layer)
    ├── 业务实体: 会员、商品、订单
    ├── 业务关系: 会员-购买-商品
    └── 业务术语: 金牌会员 = 会员等级='gold'

L2: Schema 层 (Physical Schema Layer)
    ├── 数据库表: members, products, orders
    ├── 字段: members.level, orders.total_amount
    └── 外键: orders.member_id → members.id
```

#### 预期效果

- 用户用业务术语提问 → 通过 L1 映射到 L2 → 正确生成 SQL
- 解决 P10 SEDE 揭示的真实世界语义鸿沟问题

### G2: Schema Linking 与知识召回割裂 ⭐⭐⭐⭐

#### 问题定义

**两阶段独立**：现有方法分别处理 Schema Linking 和知识推理，无法共享信息。

```
现有流程:
1. Schema Linking → 召回 Schema 项
2. 知识推理 → 推理业务知识
3. SQL 生成 → 独立使用两者

问题: 两阶段信息不共享，可能召回不相关的 Schema 或知识
```

#### 现有方法局限

| 论文 | 方法 | 局限 |
|------|------|------|
| P01-P05 | Schema Linking 方法 | 仅召回 Schema，不召回知识 |
| P13 StructGPT | 迭代推理 | Reading-Reasoning 串行，非联合 |
| P07 DataFactory | 双团队协作 | 独立协作，非联合召回 |

#### 技术可行性分析

| 技术组件 | 现有基础 | 可借鉴论文 | 可行性 |
|---------|---------|-----------|--------|
| 图谱联合表示 | 异构图神经网络 | P08 RAT-SQL, P03 DCG-SQL | ✅ 高 |
| 联合召回算法 | 多源检索、图游走 | P02 SchemaGraphSQL 图算法 | ✅ 高 |
| 元路径推理 | 知识图谱元路径 | 知识图谱推理方法 | ✅ 高 |

#### 解决方案思路

**联合召回框架**：

```
输入: 用户问题 q
输出: 相关 Schema 项 S* + 相关业务知识 K*

步骤:
1. 问题编码 → 嵌入向量 e_q
2. 图谱游走 → 从 Ontology 实体出发
3. 元路径约束 → 沿着语义正确的路径游走
4. 到达 Schema → 同时获取 Schema 项和业务知识
5. 最小完备集 → 提取必需的 Schema + 知识
```

#### 预期效果

- 一次召回同时获取 Schema 和知识
- 减少信息冗余，保证召回完整性

### G3: 缺乏结构约束机制 ⭐⭐⭐⭐

#### 问题定义

**无语义约束**：现有 Schema Linking 方法缺乏语义正确性验证机制。

```
示例：
问题: "查询每个会员的订单数"
错误召回: members, products, orders
正确召回: members, orders

问题: products 表虽然与 orders 有外键关系，但不是问题所需的
```

#### 现有方法局限

| 论文 | 方法 | 局限 |
|------|------|------|
| P02 SchemaGraphSQL | 最短路径 | 仅考虑连接性，不考虑语义 |
| P04 RSL-SQL | 双向验证 | 依赖 LLM，成本高 |
| P09 RESDSQL | Skeleton 约束 | 仅约束 SQL 结构，不约束 Schema 选择 |

#### 技术可行性分析

| 技术组件 | 现有基础 | 可借鉴论文 | 可行性 |
|---------|---------|-----------|--------|
| 元路径定义 | 知识图谱元路径 | Meta-path 挖掘方法 | ✅ 高 |
| 路径约束推理 | 图算法 | P02 SchemaGraphSQL | ✅ 高 |
| 语义验证 | LLM 验证 | P04 RSL-SQL | ⚠️ 中（依赖 LLM）|

#### 解决方案思路

**元路径约束**：

```
元路径示例:
Member → Purchase → Product
  语义: 会员购买商品

Schema 映射:
Member → members 表
Purchase → orders 表
Product → products 表

约束:
如果问题涉及"会员购买商品"，则必须同时召回 members, orders, products
如果问题仅涉及"会员的订单"，则只需召回 members, orders
```

#### 预期效果

- 通过元路径语义约束 Schema 选择
- 减少无关 Schema 召回，提高精确度

### G4: 多智能体成本高 ⭐⭐⭐

#### 问题定义

**高成本**：现有方法依赖多次 LLM 调用或多智能体协作，延迟高、成本大。

```
成本对比:
P01 LinkAlign: 多轮检索 + 多智能体辩论 → 10+ 次 LLM 调用
P05 MAG-SQL: 4 个 Agent 协作 → 5+ 次 LLM 调用
P07 DataFactory: Database + KG 双团队 → 8+ 次 LLM 调用
```

#### 现有方法局限

| 论文 | 方法 | 成本 |
|------|------|------|
| P01 LinkAlign | 多智能体辩论 | 高延迟、高成本 |
| P05 MAG-SQL | 4 个 Agent | 多次 LLM 调用 |
| P07 DataFactory | 双团队协作 | 协作开销大 |

#### 技术可行性分析

| 技术组件 | 现有基础 | 可借鉴论文 | 可行性 |
|---------|---------|-----------|--------|
| 图谱结构化推理 | 知识图谱推理 | 图算法、GNN | ✅ 高 |
| 减少依赖 LLM | 规则 + 图谱 | P02 SchemaGraphSQL 零样本 | ✅ 高 |
| 缓存机制 | 向量检索缓存 | 检索增强方法 | ✅ 高 |

#### 解决方案思路

**结构化推理替代 LLM 协作**：

```
现有方法 (高成本):
LLM 辩论 → Schema 选择 → LLM 验证 → 最终 Schema

本研究方法 (低成本):
图谱游走 → 元路径约束 → 直接得到 Schema

对比:
- 无需多智能体辩论
- 无需多次 LLM 调用
- 结构化推理保证语义正确
```

#### 预期效果

- 减少 50%+ 的 LLM 调用
- 降低延迟，提高响应速度

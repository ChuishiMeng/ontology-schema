## 1. 核心贡献

C3 提出了一个**系统性的零样本 Text-to-SQL 框架**，从模型输入、模型偏差、模型输出三个维度进行优化，由三个关键组件构成：

| 组件 | 对应维度 | 核心思想 |
|------|---------|---------|
| **Clear Prompting (CP)** | 模型输入 | 清晰的 Prompt 布局 + 基于 ChatGPT 的 Schema Linking |
| **Calibration with Hints (CH)** | 模型偏差 | 通过历史对话植入去偏提示，校正 ChatGPT 的 SQL 生成偏差 |
| **Consistent Output (CO)** | 模型输出 | 基于执行结果的自一致性投票，克服 LLM 输出随机性 |

**核心卖点**：
- 零样本设置下达到 Spider 测试集 **82.3% 执行准确率**（提交时零样本 SOTA）
- 每次查询仅使用约 **1,000 tokens**（DIN-SQL 需 10,000+ tokens）
- 超越所有微调方法（RESDSQL 79.9%），仅用 GPT-3.5 而非 GPT-4

## 2. 技术方法详解

### 2.1 Clear Prompting (CP) — 清晰提示

CP 分为两个子模块：**Clear Layout** 和 **Clear Context**。

#### 2.1.1 Clear Layout（清晰布局）

对比两种 Prompt 布局风格：

**Complicated Layout（传统 PLM 风格）**：
```
Complete sqlite SQL query only and with no explanation.
Which States have both owners and professionals living there? 
Sqlite SQL tables, with their properties: owners : owners.state, owners.owner_id ...
```

**Clear Layout（C3 采用）**：
```
### Complete sqlite SQL query only and with no explanation,
### and do not select extra columns that are not explicitly requested in the query.
### Sqlite SQL tables, with their properties:
#
# Owners (state, owner_id, first_name)
# Professionals (state, first_name, last_name)
# Dogs (owner_id, dog_id, name, age)
#
### Which States have both owners and professionals living there?
SELECT
```

**关键设计**：
- 用 `###` 分隔 Instruction / Context / Question 三个区域
- 用 `#` 标注 Schema 信息
- 额外加入指令："do not select extra columns that are not explicitly requested"
- **效果**：Clear Layout 比 Complicated Layout 高 **7.0%** 执行准确率

#### 2.1.2 Clear Context — 基于 ChatGPT 的 Schema Linking

**问题**：直接放入完整数据库 Schema 会导致：(1) 引入无关列，误导生成；(2) Token 消耗过大

**解决方案**：两步零样本 Schema Linking

**Step 1: Table Recall（表召回）**
- 设计零样本 Prompt 让 ChatGPT 按相关性排序所有表
- 包含三个子步骤：排序 → 检查是否遗漏 → 输出列表格式
- 使用 **Self-Consistency**：生成 10 组召回结果，取 top-4 表中出现最频繁的
- **Table AUC = 0.9588**

**Step 2: Column Recall（列召回）**
- 在召回的表中，对每个表的列按相关性排序
- 强调：与问题词匹配的列和外键列优先排前
- 使用 **Self-Consistency**：每表生成 10 组列排序，取出现最频繁的 top-5 列
- **Column AUC = 0.9833**

**Step 3: Foreign Key 信息**
- 将召回表之间的外键关系加入 Context
- 格式：`# concert.stadium_id = stadium.stadium_id`

### 2.2 Calibration with Hints (CH) — 偏差校准

通过分析 ChatGPT 生成 SQL 的错误模式，识别出两类系统性偏差：

#### Bias 1: 保守选列偏差
- ChatGPT 倾向于选择与问题相关但**非必要**的列
- 典型案例：问"哪个年份有最多的X"时，ChatGPT 会同时 SELECT Year 和 COUNT(*)，但标准答案只需 Year

#### Bias 2: SQL 关键字误用偏差
- ChatGPT 倾向于使用 LEFT JOIN、OR、IN，但经常用错
- 导致执行结果包含多余值

#### 校准策略：通过历史对话植入 Hints

CH 利用 ChatGPT 的**多轮对话能力**，在 System/User/Assistant 历史消息中植入去偏提示：

**Hint 1（针对 Bias 1）**：
```
Tips 1: 
Question: Which A has most number of B?
Gold SQL: select A from B group by A order by count(*) desc limit 1;
Notice that the Gold SQL doesn't select COUNT(*) because the question only wants to know the A...
```

**Hint 2（针对 Bias 2）**：
```
Tips 2:
Don't use "IN", "OR", "LEFT JOIN" as it might cause extra results, 
use "INTERSECT" or "EXCEPT" instead, and remember to use "DISTINCT" or "LIMIT" when necessary.
```

**技术实现**：通过 ChatGPT API 的 `messages` 参数，构造 System → User(Hint1) → Assistant(确认) → User(Hint2) → Assistant(确认) → User(实际查询) 的多轮对话。

### 2.3 Consistent Output (CO) — 一致性输出

**问题**：即使用了 CP + CH，ChatGPT 的输出仍然不稳定。实验发现：30 次独立实验中，只有不到 65% 的 SQL 能始终正确。

**解决方案**：基于执行结果的自一致性（Execution-based Self-Consistency）

**流程**：
1. 单次 API 调用生成 **20 条** SQL 查询（利用 `n=20` 参数）
2. 在数据库上执行所有 SQL，收集执行结果
3. 去除报错的 SQL
4. 按**执行结果**对 SQL 进行聚类
5. 选择最大聚类中的一条 SQL 作为最终输出

**关键洞察**：
- 同一个问题可以有多种不同写法的正确 SQL
- 不比较 SQL 文本，而比较**执行结果**
- 投票机制能有效抵消 LLM 的随机性

## 3. 实验结果

### 3.1 主实验（Spider 数据集）

| 方法 | 类型 | Dev | Test |
|------|------|-----|------|
| ChatGPT-SQL + ChatGPT | 零样本 | 72.3 | — |
| RATSQL + GAP + NatSQL | 微调 | 75.0 | 73.3 |
| T5-3B + PICARD | 微调 | 79.3 | 75.1 |
| Graphix-3B + PICARD | 微调 | 81.0 | 77.6 |
| SC-Prompt + T5-3B | 微调 | 81.1 | — |
| RESDSQL-3B + NatSQL | 微调 | 84.1 | 79.9 |
| DIN-SQL + GPT-4 | 少样本 | 82.8 | 85.3 |
| **C3 + ChatGPT (本文)** | **零样本** | **81.8** | **82.3** |

**关键发现**：
- 测试集上超越所有微调方法 2.4%（vs RESDSQL 79.9%）
- 比零样本基线 ChatGPT-SQL 提升 9.5%（dev）
- 仅用 GPT-3.5，token 消耗约为 DIN-SQL (GPT-4) 的 10%
- 提交时 Spider 排行榜第 2 名

### 3.2 消融实验

| 配置 | Dev Acc | 变化 |
|------|---------|------|
| C3 完整 | 81.8 | — |
| 去掉 Table/Column Recall | 79.5 | ↓ 2.3 |
| 去掉 Foreign Keys | 79.2 | ↓ 2.6 |
| 去掉 Calibration (CH) | 80.3 | ↓ 1.5 |
| 去掉 Self-Consistency (CO) | 80.5 | ↓ 1.3 |

**各组件贡献排序**：Foreign Keys (2.6%) > Schema Linking (2.3%) > Calibration (1.5%) > Self-Consistency (1.3%)

### 3.3 Schema Linking 质量

| 指标 | AUC |
|------|-----|
| Table Recall | 0.9588 |
| Column Recall | 0.9833 |

### 3.4 潜力分析

30 次独立实验中，如果任一次正确即算对：
- ChatGPT-SQL：85.7%（单次仅 72.3%，gap = 13.4%）
- CP：~88%
- CP + CH：~89%

说明 ChatGPT 的潜力远超单次表现，Self-Consistency 能有效挖掘这一潜力。

### 3.5 错误分析

| 类别 | ChatGPT-SQL 占比 | C3 占比 | 变化 |
|------|-----------------|--------|------|
| Schema-linking (Extra columns) | 较高 | ↓ 11% | CH 校准 Bias 1 的效果 |
| Nested (Set op) | 中等 | 略降 | CH 校准 Bias 2 |
| Other (Distinct) | 中等 | 略降 | CH 校准 Bias 2 |
| Join (Wrong tables/columns) | 较高 | 显著降 | CP 的 Schema Linking 效果 |
| **总错误数** | **286** | **188** | **↓ 34%** |

## 4. Schema 处理策略（零样本设置下的关键创新）

C3 的 Schema 处理是其核心创新之一，完全**不依赖训练数据**：

### 4.1 处理流程

```
原始 Schema（完整数据库）
    ↓ Table Recall（ChatGPT + Self-Consistency × 10）
选出 Top-4 相关表
    ↓ Column Recall（ChatGPT + Self-Consistency × 10）
每表选出 Top-5 相关列
    ↓ 加入 Foreign Key 关系
构造精简 Schema → 放入 Prompt Context
```

### 4.2 关键设计决策

1. **用 LLM 做 Schema Linking**：而非传统的字符串匹配或嵌入相似度
2. **Self-Consistency 保证稳定性**：10 次生成取投票结果
3. **保留外键信息**：外键对 JOIN 操作至关重要（去掉后降 2.6%）
4. **控制召回数量**：top-4 表 × top-5 列，控制 token 在 ~1000 以内

### 4.3 与传统 Schema Linking 对比

| 特性 | 传统方法 | C3 |
|------|---------|-----|
| 是否需要训练 | ✅ 需要标注数据 | ❌ 零样本 |
| Schema 表示 | 图结构 / 嵌入 | 文本格式 |
| 匹配策略 | 字符串匹配 + 嵌入相似度 | LLM 语义理解 |
| 外键处理 | 编码到图中 | 文本形式附加 |
| 跨域泛化 | 受训练域限制 | 天然跨域 |

## 5. 方法论洞察

### 5.1 系统性思维

C3 的核心贡献不仅是具体技术，更是提出了一个**系统性框架**：

```
Text-to-SQL 优化 = 输入优化 (CP) + 偏差校准 (CH) + 输出优化 (CO)
```

这个三维框架具有普适性，可以应用到其他 LLM 任务。

### 5.2 "Prompt as Knowledge" 范式

CH 组件展示了一种有趣的知识注入方式：
- 不需要微调或 few-shot 示例
- 通过**构造虚拟历史对话**来植入领域知识
- 本质上是一种 "Prompt Engineering as Calibration"

### 5.3 效率优势

| 方法 | 模型 | Token/Query | API 成本 |
|------|------|------------|---------|
| DIN-SQL | GPT-4 | 10,000+ | 极高 |
| DAIL-SQL | GPT-4 | 数千 | 高 |
| **C3** | **GPT-3.5** | **~1,000** | **低** |

## 6. 局限性

1. **依赖 GPT-3.5 的特定行为**：Hints 是针对 ChatGPT 的特定偏差设计的，换模型可能不适用
2. **Self-Consistency 需要执行数据库**：CO 需要真实执行 SQL，在某些场景下不可行
3. **Schema Linking 有天花板**：AUC 虽高但非 100%，错误的 Schema Linking 会导致不可恢复的错误
4. **仅评估 Spider**：缺少在 BIRD、WikiSQL 等其他数据集上的验证
5. **Hints 设计依赖人工分析**：需要手动分析错误模式来设计去偏提示

## 7. 与本研究（Ontology-Schema）的关系

### 7.1 直接相关点

| 维度 | C3 的做法 | 与本研究的关联 |
|------|----------|--------------|
| **Schema 表示** | 文本格式（表名 + 列名 + 外键） | 本研究探索 Ontology 增强的语义 Schema 表示，可作为 C3 Schema 表示的升级 |
| **Schema Linking** | LLM 零样本召回 | 本研究的本体知识可提供更精准的语义约束，增强 Schema Linking |
| **Prompt 设计** | Clear Layout + Calibration | 本研究可借鉴其系统性 Prompt 设计思路 |
| **零样本能力** | 完全不需要训练数据 | 本研究同样关注跨域泛化能力 |

### 7.2 可借鉴的技术

1. **Schema Linking 的 Self-Consistency 思路**：用于本体映射的稳定性增强
2. **偏差校准 (CH) 范式**：可用于校准 LLM 在本体推理中的常见错误
3. **执行反馈的自一致性**：可扩展到本体约束验证
4. **Token 效率控制**：Schema 精简策略值得借鉴

### 7.3 C3 的不足 = 本研究的机会

| C3 的不足 | 本研究的可能突破 |
|----------|----------------|
| Schema 仅用文本表示，缺乏语义 | Ontology 提供丰富的语义关系（is-a, part-of, domain/range） |
| Schema Linking 依赖表面词匹配 | 本体推理可支持深层语义匹配 |
| 外键是唯一的关系信息 | 本体可提供业务规则、约束、层级关系等 |
| Hints 针对特定模型手工设计 | 本体知识可自动生成领域特定的约束提示 |

## 1. 核心贡献和创新点

### 1.1 核心定位
CodeS 是一系列**完全开源的 Text-to-SQL 专用语言模型**，基于 StarCoder 进行增量预训练，参数量从 1B 到 15B。其核心目标是解决闭源 LLM（如 GPT-4）在 Text-to-SQL 领域的三大局限：
- **L1**: 闭源模型架构不透明，无法针对性优化
- **L2**: API 调用存在数据隐私风险
- **L3**: 参数量巨大（如 GPT-3.5 的 175B），推理成本高

### 1.2 四大创新贡献

**① 增量预训练方法 + SQL-centric 语料库**
- 收集 21.5GB 专用语料：SQL 数据 (11GB) + NL-to-code 数据 (6GB) + NL 数据 (4.5GB)
- 在 StarCoder 基础上进行增量预训练，显著提升 SQL 生成能力
- 开源了预训练语料库

**② 全面的数据库 Prompt 构建策略**
- **Schema Filter**: 基于神经网络分类器过滤无关表和列
- **Value Retriever**: 粗到细 (BM25 + LCS) 的数据库值检索
- **丰富的元数据**: 列类型、注释、代表值、主外键

**③ 双向数据增强 (Bi-directional Data Augmentation)**
- **Question-to-SQL 方向**: 收集真实用户问题 → GPT-3.5 扩展
- **SQL-to-Question 方向**: 模板填充 + GPT-3.5 改写
- 仅需 30 个标注样本即可适配新领域

**④ 小模型超越大模型**
- CodeS (1B-15B) 比 GPT-4/ChatGPT (175B+) 小 10x-100x
- 在多个基准上达到或超越 SOTA

## 2. 技术方法细节

### 2.1 增量预训练

**基座模型**: StarCoder 系列
- CodeS-15B ← StarCoder-15B
- CodeS-1B/3B/7B ← StarCoderBase-1B/3B/7B

**预训练语料构成**:

| 数据类型 | 大小 | 来源 | 训练轮数 |
|----------|------|------|---------|
| SQL-related | 11GB | StarCoder 预训练语料的 SQL 部分 | 2 epochs |
| NL-to-code | 6GB | CoNaLa, StaQC, CodeAlpaca-20k, Jupyter notebooks, **NL-SQL-458K (自建)** | 1 epoch |
| NL-related | 4.5GB | Alpaca-cleaned, Unnatural-instructions, UltraChat | 1 epoch |

**NL-SQL-458K 构建方法**:
- 用正则从 The Pile, The Stack, GitHub Code 中提取 SELECT 查询
- 过滤语法错误 → 458K SQL
- GPT-3.5 生成对应的自然语言问题

**训练配置**:
- 优化器: AdamW (β1=0.9, β2=0.95, ε=10⁻⁸)
- 学习率: 5e-5, cosine decay
- Batch size: 4M tokens
- 混合精度: BF16 + DeepSpeed Zero-3
- FlashAttention-2
- 最大上下文: 8192 (1B/3B/7B) / 6144 (15B)
- 训练时长: 1.5天(1B) / 3天(3B) / 8天(7B) / 16天(15B)

### 2.2 数据库 Prompt 构建

**Schema Filter**:
- 训练一个 schema item classifier（基于 RESDSQL 的方法）
- 预测每个表和列与用户问题的相关性得分
- 保留 top-k₁ 个表（默认6），每表 top-k₂ 个列（默认10）
- AUC 表现: Spider (Table 0.991, Column 0.993), BIRD (Table 0.966, Column 0.943)

**Value Retriever (粗到细)**:
- **粗筛**: BM25 索引（Lucene/Pyserini），快速检索数百个候选值
- **细筛**: LCS（最长公共子串）算法精确匹配
- 解决大规模数据库（如 BIRD Donor 库 1.165 亿值）的效率问题

**元数据包含**:
1. **列数据类型** (INTEGER, REAL, TEXT 等)
2. **列注释/Comments** (处理歧义 schema，如 `w1st` → "word id of the first word")
3. **代表性列值** (每列 2 个 DISTINCT 非空值)
4. **主键和外键** (指导 JOIN ON 生成)

### 2.3 双向数据增强

**Question-to-SQL 方向** (保真性):
1. 收集少量真实用户问题
2. 人工标注 SQL（约 30 个样本）
3. GPT-3.5 两阶段扩展:
   - Stage-1: 基于已有问题生成新问题
   - Stage-2: 为新问题生成对应 SQL

**SQL-to-Question 方向** (覆盖性):
1. 从 Spider 提取 75 个通用 (question, SQL) 模板
2. 用新数据库的表/列/值填充模板
3. GPT-3.5 改写模板化问题使其更自然

### 2.4 模型使用方式

**监督微调 (SFT)**:
- 4 epochs, batch size 128, 学习率 5e-6
- 最大上下文 4096
- Beam search 生成 4 个 SQL 候选，选第一个可执行的

**Few-shot In-Context Learning**:
- 无需微调，利用预训练能力
- **Question-pattern-aware demonstration retriever**: 
  - 同时考虑问题语义相似度和问题模式相似度
  - 用 NLTK 去除实体，提取问题模式
  - SimCSE 计算相似度
  - 避免检索结果过度偏向实体匹配

## 3. 实验结果

### 3.1 Spider 基准 (Development Set)

| 方法 | EX% | TS% |
|------|-----|-----|
| T5-3B + PICARD | 79.3 | 69.4 |
| RESDSQL-3B + NatSQL | 84.1 | 73.5 |
| Fine-tuned SQL-PaLM | 82.8 | 78.2 |
| DIN-SQL + GPT-4 | 82.8 | 76.6 |
| DAIL-SQL + GPT-4 | 83.6 | 76.2 |
| SFT Llama2-7B | 77.9 | 72.2 |
| SFT Llama2-13B | 83.4 | 78.1 |
| **SFT CodeS-3B** | **85.4** | **80.3** |
| **SFT CodeS-7B** | **84.9** | **79.4** |
| **SFT CodeS-15B** | 77.9 (可能过拟合) | 72.2 |

> ⭐ **SFT CodeS-3B 已超越 GPT-4 based 方法**（DIN-SQL, DAIL-SQL）

### 3.2 BIRD 基准

**Development Set**:

| 方法 | EX% | EX% (w/ EK) |
|------|-----|-------------|
| ChatGPT | 25.88 | 32.33 |
| GPT-4 | 41.85 | 49.15 |
| DIN-SQL + GPT-4 | 50.72 | 55.90 |
| DAIL-SQL + GPT-4 | 54.76 | 57.41 |
| SFT CodeS-7B | 57.17 | 60.37 |
| **SFT CodeS-15B** | **58.47** | **63.62** |

**Test Set**:

| 方法 | EX% | EX% (w/ EK) |
|------|-----|-------------|
| ChatGPT + CoT | 28.95 | 36.47 |
| DIN-SQL + GPT-4 | - | 55.90 |
| **SFT CodeS-7B** | 52.15 | 59.25 |
| **SFT CodeS-15B** | **52.15** | **60.37** |

> ⭐ **在 BIRD 上提升最显著**: SFT CodeS-15B 比 ChatGPT+CoT 提升 23.20% (EX)

### 3.3 鲁棒性基准

| 基准 | 之前 SOTA | SFT CodeS-7B |
|------|-----------|-------------|
| Spider-Syn (TS%) | 67.4 | **70.0** (+2.6) |
| Spider-Realistic (TS%) | 73.2 | **77.2** (+4.0) |

| Dr.Spider (Global EX%) | 74.9 | **75.0** (+0.1) |

### 3.4 真实场景 (新领域适配)

**Bank-Financials** (金融，4表，最大表65列):

| 方法 | EX% | HE% |
|------|-----|-----|
| 3-shot GPT-3.5 + comments | 57.1 | 84.6 |
| DIN-SQL + GPT-4 | 26.4 | 79.1 |
| 3-shot CodeS-7B | 61.5 | 78.0 |
| **SFT CodeS-7B (aug. data)** | **71.4** | **85.7** |

### 3.5 推理效率

| 模型 | 推理时间/样本 | GPU 显存 (FP16) |
|------|-------------|----------------|
| DIN-SQL + GPT-4 | ~60 秒 | - |
| SFT CodeS-1B | 0.6 秒 | 10GB |
| SFT CodeS-3B | 0.9 秒 | 13GB |
| SFT CodeS-7B | 1.1 秒 | 20GB |
| SFT CodeS-15B | 1.5 秒 | 35GB |

> ⭐ CodeS-7B 推理速度是 DIN-SQL+GPT-4 的 **55 倍**

### 3.6 消融实验关键发现

| 组件 | 移除后影响 |
|------|-----------|
| Schema Filter | 性能下降 + 生成速度变慢 |
| Value Retriever | BIRD 上下降明显（对 predicate 生成至关重要） |
| Comments | BIRD 上下降显著（歧义 schema 多） |
| 代表性值 | 中等下降 |
| 主外键 | 中等下降（影响 JOIN ON 生成） |
| 列数据类型 | 影响较小 |
| Pattern similarity | Spider 下降明显，BIRD 影响较小 |

## 4. 与本研究 (Ontology-Schema) 的关系

### 4.1 直接关联

| 维度 | 关联点 |
|------|--------|
| **Schema 理解** | CodeS 的 schema filter + 元数据策略 = 对数据库结构的深度理解，与 ontology-based schema 理解高度相关 |
| **歧义消解** | CodeS 用 comments 解决歧义 schema（如 `w1st` → "word id"），这本质上是一种轻量级的 ontology annotation |
| **领域适配** | 双向数据增强方法可借鉴用于 ontology-schema 的新领域扩展 |
| **开源可控** | 完全开源，可作为下游研究的基座或对比系统 |

### 4.2 Schema 表示的启发

CodeS 的 prompt 构建策略实质上是一种**扁平化的 schema 语义表示**：
- 表名 + 列名 + 类型 + 注释 + 代表值 + 主外键
- 这种表示**缺乏显式的语义层次和关系结构**
- Ontology-based schema 可以提供更丰富的语义信息：
  - 概念层次（is-a 关系）
  - 属性约束
  - 领域知识图谱
  - 跨表语义关联

### 4.3 研究空白 → 机会

1. **Schema 理解的天花板**: CodeS 依赖 comments 和代表值来理解 schema，但这仍是"表面语义"。Ontology 可提供**深层语义理解**
2. **跨库泛化**: CodeS 在新领域仍需数据增强和微调。Ontology-based 方法有望通过统一语义层实现**零样本跨库泛化**
3. **复杂推理**: CodeS 在 BIRD 上表现远不如 Spider，说明复杂推理仍有提升空间。Ontology 的推理规则可辅助复杂 SQL 生成

## 5. 作为 Baseline 的价值

### 5.1 为什么适合作为 Baseline

| 理由 | 说明 |
|------|------|
| ✅ **SOTA 代表性** | 2024 年开源 Text-to-SQL 的 SOTA，代表了 SFT 路线的最强水平 |
| ✅ **完全开源** | 代码、模型、数据全部开源，可完全复现 |
| ✅ **多规模可选** | 1B-15B 四个版本，可根据计算资源选择 |
| ✅ **覆盖面广** | Spider, BIRD, 鲁棒性基准, 真实场景均有评估 |
| ✅ **高效推理** | 7B 模型仅需 20GB 显存，1.1 秒/样本 |

### 5.2 作为 Baseline 的使用建议

1. **直接对比**: 在 Spider/BIRD 上与 CodeS 的公开结果对比
2. **Schema 增强实验**: 用 ontology-enriched schema 替换 CodeS 的 prompt，测试性能差异
3. **消融对比**: 
   - CodeS (原始 prompt) vs CodeS (+ ontology annotation)
   - 验证 ontology 信息的增量价值
4. **领域适配对比**: 在新领域上对比 CodeS 双向增强 vs ontology-guided 增强

### 5.3 局限性（作为 Baseline 需注意）

- 基于 StarCoder (2023)，基座模型已非最新
- 15B 最大，无法与 GPT-4 级别模型直接对比
- Spider test set 结果缺失（硬件限制）
- 训练依赖 8×A800 GPU，复现门槛较高

## 📝 阅读笔记

**优点**:
1. 系统性工程：从预训练→prompt构建→数据增强→部署，形成完整 pipeline
2. 小模型大性能：3B 模型超越 GPT-4 based 方法
3. 实用性强：推理快、显存需求低、可本地部署
4. 双向数据增强方法新颖，30 个标注样本即可适配新领域

**不足**:
1. Schema 理解仍停留在"字面+注释"层面，缺乏深层语义建模
2. 基座模型 StarCoder 非最新（已有 StarCoder2 等后续版本）
3. 在 BIRD 上 EX 最高 ~58%，复杂场景仍有很大提升空间
4. VES 指标不稳定（受硬件影响），实验说服力有限

**对本研究的启示**:
- CodeS 代表了"工程优化路线"的极限，要超越它需要从**语义理解**层面突破
- Ontology-schema 方法如果能在 BIRD 的歧义 schema 上展现优势，将是有力的差异化贡献
- 可以考虑将 ontology 信息注入 CodeS 的 prompt 中作为实验验证

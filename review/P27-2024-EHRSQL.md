## 一、核心贡献与创新点

### 1.1 研究动机

现有 EHR QA 数据集（MIMICSQL、emrKBQA）远未满足医院实际需求：
- **MIMICSQL**：基于预定义规则自动生成问题，仅覆盖 5 张表，SQL 简单
- **emrKBQA**：范围局限于门诊患者检验结果

EHRSQL 的核心动机：**从真实医院调研出发，构建反映实际需求的 Text-to-SQL 基准**。

### 1.2 三大创新

| 创新点 | 内容 | 意义 |
|--------|------|------|
| **① 真实需求驱动** | 对韩国大学医院 222 名医护人员进行调查，收集 1,742 条真实问题 | 问题来源于不了解数据库 Schema 的真实用户（?Schema=✓），更贴近实际 |
| **② 时间敏感问题** | 系统化分类时间表达（绝对/相对/混合 × 单位 × 区间类型），93.2% 的查询涉及时间列 | 医疗领域时间是核心维度，远超 Spider (12.7%) 和 MIMICSQL (26.6%) |
| **③ 不可回答问题检测** | 首次在 Text-to-SQL 中引入不可回答问题（验证集和测试集中占 33%） | 提出"可信语义解析"(Trustworthy Semantic Parsing) 新任务范式 |

### 1.3 数据集规模

| 指标 | EHRSQL | Spider | MIMICSQL |
|------|--------|--------|----------|
| 样本量 | 24K | 8K | 10K |
| 数据库数 | 2 (MIMIC-III + eICU) | 160 | 1 |
| 表数/DB | 13.5 | 5.1 | 5 |
| 行数/表 | 108K | 2K | 7K |
| 嵌套层数/Q | 2.7 | 1.2 | 1.0 |
| 时间列使用率 | **93.2%** | 12.7% | 26.6% |
| 用户不知Schema | ✓ | ✗ | ✗ |
| 不可回答问题 | ✓ | ✗ | ✗ |

## 二、业务术语问题的展示 ⭐

### 2.1 医疗领域术语鸿沟的典型案例

EHRSQL 深刻展示了**业务用户术语 vs 数据库 Schema 术语**的鸿沟：

#### 案例1：医护人员的自然提问 vs 数据库结构

| 医护人员提问（业务术语） | 需要映射的 Schema 元素 | 鸿沟类型 |
|--------------------------|----------------------|----------|
| "患者的**体温**最后一次测量是什么时候？" | `chartevents.itemid` → `d_items.label = 'temperature c (calc)'` | 术语映射：体温 → temperature c (calc) |
| "患者的**动脉血压（收缩压）**是多少？" | `chartevents.itemid` → `d_items.label = 'arterial bp [systolic]'` | 术语映射：收缩压 → arterial bp [systolic] |
| "**非侵入性机械通气**的平均总住院费用是多少？" | `cost` 表 + `procedures_icd` 表 + 过程名映射 | 专业术语 + 跨表关联 |
| "**低血压NOS**诊断后最常开的 5 种药物" | `diagnoses_icd` → `d_icd_diagnoses.long_title = 'hypotension nos'` | 疾病编码映射 |
| "计算**五年生存率**最低的五种诊断" | 需要从入院/出院/死亡时间计算 | 隐性逻辑：生存率需要多步推理 |

#### 案例2：Schema 的层次结构导致的复杂性

论文 Figure 2 展示了 EHR Schema 的层级结构：

```
admissions (患者入院) → icustays (ICU住院) → chartevents (监测记录)
                                              ↓
                                          d_items (数据字典)
```

一个简单的"患者体温"查询需要：
1. `admissions` → 找到 `subject_id` 对应的 `hadm_id`
2. `icustays` → 通过 `hadm_id` 找到 `icustay_id`
3. `chartevents` → 通过 `icustay_id` 找到监测记录
4. `d_items` → 通过 `itemid` 映射到 `label = 'temperature c (calc)'`

**平均每个查询涉及 2.4 张表、2.7 层嵌套**，远超一般 Text-to-SQL 基准。

### 2.2 不可回答问题的业务术语特征

| 不可回答类型 | 示例 | 原因 |
|-------------|------|------|
| 需要外部知识 | "造影动脉造影-腿部治疗期间不应使用什么药物？" | 需要医学禁忌知识，不在 DB 中 |
| 超出 Schema | "患者 73652 下一次最早的就诊是什么时候？" | 数据库不存储未来预约 |
| 其他院区信息 | "患者在其他科室的诊断" | eICU 不包含院际数据 |
| 患者个人信息 | "患者的家属联系方式" | 去标识化后不可得 |

**核心启示**：真实用户不知道数据库能回答什么、不能回答什么。**不可回答问题检测**是部署可信 Text-to-SQL 系统的必要条件。

### 2.3 SQL 标注中的术语假设问题

论文明确指出 SQL 标注过程中面临大量**假设决策**：
> "SQL labeling required numerous assumptions (e.g., which medical event is mapped to which column in the database, the choice of rank functions in SQL, the occasion of using DISTINCT, etc.)"

4 名研究生耗时 **5 个月**完成标注，经过反复修订。这说明即使是领域专家，业务术语到 Schema 的映射也需要大量人工判断。

## 三、实验设置与结果

### 3.1 任务定义：可信语义解析（Trustworthy Semantic Parsing）

```
输入: 自然语言问题 Q
      ↓
模型生成 SQL + 置信度分数
      ↓
置信度 < 阈值? → 执行 SQL → 返回答案
置信度 ≥ 阈值? → 拒绝回答
```

### 3.2 评估指标体系

| 指标 | 含义 | 公式 |
|------|------|------|
| **F1_ans** | 可回答/不可回答分类的 F1 | 正确识别可回答问题的 P/R 的调和平均 |
| **P_exe** | 执行精度 | 正确回答数 / 预测为可回答数 |
| **R_exe** | 执行召回 | 正确回答数 / 实际可回答数 |
| **F1_exe** | 执行 F1（核心指标） | P_exe 和 R_exe 的调和平均 |

**关键设计**：F1_exe 同时评估两个能力——① 拒绝不可回答问题 ② 生成正确 SQL。

### 3.3 基线模型与置信度策略

**模型**：T5-base、T5-base + Schema 序列化

**置信度阈值策略**：
| 策略 | 方法 | 效果 |
|------|------|------|
| None | 不拒绝，所有问题都尝试回答 | F1_exe ≈ 76-77% |
| Clustering-based | K-means (k=2) 聚类最大熵值 | F1_exe ≈ 82-87% |
| **Percentile-based** | 取验证集 67th 百分位最大熵 | **F1_exe ≈ 89-93%** |

### 3.4 主要结果

#### MIMIC-III 测试集（Percentile-based 阈值）

| 模型 | F1_ans | P_exe | R_exe | F1_exe |
|------|--------|-------|-------|--------|
| T5 | 91.4 | 88.8 | 91.1 | **89.9** |
| T5 + Schema | 90.6 | 88.4 | 89.7 | 89.1 |

#### eICU 测试集（Percentile-based 阈值）

| 模型 | F1_ans | P_exe | R_exe | F1_exe |
|------|--------|-------|-------|--------|
| T5 | 90.8 | 87.5 | 91.6 | **89.5** |
| T5 + Schema | 90.9 | 84.8 | 93.5 | 88.9 |

#### 关键发现

1. **Percentile-based 阈值最优**：利用先验知识（33% 不可回答）效果最好
2. **Schema 序列化无显著提升**：单数据库场景下，Schema 信息未被有效利用（与 SEDE 论文发现一致）
3. **跨领域迁移惨淡**：Spider SOTA 模型 GAP 在 EHRSQL 上仅 **4.7%** 执行准确率（vs MIMICSQL 16.4%），且 93% 的 EHRSQL 查询无法被 Spider 解析器处理

## 四、与本研究的关系 ⭐⭐

### 4.1 直接相关：业务术语与 Schema 语义鸿沟

EHRSQL 是**业务术语与 Schema 语义鸿沟**问题的绝佳案例证据：

| 本研究核心概念 | EHRSQL 中的体现 |
|---------------|-----------------|
| **语义鸿沟** | 医护人员用"体温"提问 → Schema 中是 `d_items.label='temperature c (calc)'` |
| **隐性业务逻辑** | "五年生存率" → 需要从入院/出院/死亡时间多步计算 |
| **业务知识碎片化** | 不可回答问题的存在说明业务知识分散在 Schema 之外 |
| **跨表推理** | 平均 2.4 表/查询，2.7 层嵌套，需要理解层级关系 |

### 4.2 EHRSQL 的局限 = 本研究的机会

| EHRSQL 局限 | 本研究可解决的方向 |
|------------|-------------------|
| ❌ 没有引入结构化业务知识层（Ontology） | ✅ Ontology-Schema 联合召回可显式建模术语映射 |
| ❌ 术语映射硬编码在 SQL 模板中 | ✅ Ontology 可将映射关系结构化、可复用 |
| ❌ Schema 序列化无效 | ✅ 业务知识增强的 Schema Linking 可能更有效 |
| ❌ 不可回答问题仅通过置信度检测 | ✅ Ontology 覆盖范围可直接判断可回答性 |
| ❌ SQL 标注耗时 5 个月 | ✅ Ontology 半自动构建可降低成本 |

### 4.3 可引用的关键论据

在论文中可引用 EHRSQL 作为以下论据：

1. **语义鸿沟的实证**：222 名真实医护人员的调查表明，业务用户的提问方式与 Schema 存在系统性差异
2. **跨领域迁移失败**：Spider SOTA 在 EHRSQL 上仅 4.7%，证明通用 Text-to-SQL 无法直接应用于领域场景
3. **Schema 信息不足**：T5 + Schema 未优于 T5，说明单纯序列化 Schema 不够，需要额外的业务知识层
4. **可信度需求**：医疗场景对错误零容忍，不可回答问题检测是刚需

## 五、可借鉴的评估方法

### 5.1 可信语义解析评估框架

**F1_exe 指标设计**非常值得借鉴：

```
传统评估: Execution Accuracy = 正确数 / 总数
  → 问题：假设所有问题都可回答

EHRSQL 评估: F1_exe = harmonic_mean(P_exe, R_exe)
  → 优势：同时评估"敢不敢拒绝"和"回答是否正确"
```

### 5.2 置信度拒绝机制

基于解码过程中最大熵值的拒绝策略，可迁移到 Ontology-enhanced Text-to-SQL：
- **Ontology 覆盖检测**：问题涉及的业务概念是否在 Ontology 中有映射？
- **Schema Linking 置信度**：Ontology → Schema 的映射置信度是否足够？
- **联合拒绝策略**：Ontology 覆盖 + SQL 生成置信度的双重门控

### 5.3 数据集构建方法论

| 方法 | 可借鉴之处 |
|------|-----------|
| 真实用户调研 → 问题模板 → 值填充 → 释义 | 可用于构建领域 Text-to-SQL 评估集 |
| 分离"用户不知 Schema" 的设置 | 更真实地模拟业务场景 |
| 不可回答问题仅在验证/测试集 | OOD 检测的标准设置 |
| 时间表达式的系统化分类 | 可推广到其他领域特有表达 |

### 5.4 跨领域迁移实验设计

用 Spider SOTA 模型在领域数据集上零样本测试 → 证明领域鸿沟的存在。本研究可以类似地：
1. 用通用 Text-to-SQL（无 Ontology）在企业数据集上测试 → 基线
2. 加入 Ontology → 对比提升 → 证明 Ontology 的价值

## 六、方法论细节

### 6.1 数据构建流程

```
调研（222人） → 1,742 条原始问题
    ↓
过滤 + 模板化 → 230 个问题模板（174 可回答 + 56 不可回答）
    ↓
操作值采样 → 时间模板采样 → 条件值采样
    ↓
释义生成（人工 21 条/模板 + 机器 47 条/模板）
    ↓
质量审核 → 最终 24,411 条 QA 对
```

### 6.2 SQL 标注策略

- **避免 JOIN**：由于 EHR 表极大（chartevents 3.3 亿行），使用嵌套子查询替代 JOIN
- **层级结构利用**：遵循 `admissions → icustays → chartevents` 的层级关系
- **数据字典映射**：通过 `d_items`、`d_icd_diagnoses` 等字典表实现术语映射

### 6.3 时间表达式体系

| 时间过滤类型 | 表达类型 | 单位 | 区间类型 | 示例 |
|-------------|---------|------|---------|------|
| time_filter_global | 绝对 | 天 | since | "自 2104/05/01 以来" |
| time_filter_global | 相对 | 年 | in | "去年"/"今年" |
| time_filter_within | - | 住院次 | in | "同一次住院期间" |
| time_filter_exact | 相对 | 精确 | at | "第一次"/"最后一次" |

## 七、局限性

1. **数据源单一**：问题来自一所韩国大学医院，可能不代表全球各类医院
2. **释义缺乏医学术语**：使用通用领域释义工具，未包含重度医学术语表达
3. **数据库数量有限**：仅 2 个 EHR 数据库，不足以训练跨数据库泛化模型
4. **基线模型较弱**：仅用 T5-base，未测试 GPT-4 等大模型（论文发表于 2022 年）

## 八、总结与评价

### 整体评价：⭐⭐⭐⭐ (4/5)

| 维度 | 评分 | 评语 |
|------|------|------|
| 问题重要性 | ⭐⭐⭐⭐⭐ | 首次系统性揭示医疗 Text-to-SQL 的实际需求 |
| 方法创新性 | ⭐⭐⭐⭐ | 可信语义解析任务定义新颖，不可回答问题检测有价值 |
| 实验充分性 | ⭐⭐⭐ | 基线模型较弱，但数据集分析详尽 |
| 对本研究价值 | ⭐⭐⭐⭐⭐ | **直接证明业务术语-Schema 语义鸿沟的存在** |

### 核心启示

> EHRSQL 以医疗领域为案例，深刻揭示了一个普适性问题：**当真实业务用户（不了解数据库 Schema）提出问题时，业务术语与 Schema 术语之间存在系统性的语义鸿沟**。这正是本研究（Ontology-Schema 联合召回框架）要解决的核心问题。EHRSQL 的数据构建方法论（真实用户调研 + 不可回答问题）和评估框架（F1_exe + 置信度拒绝）对本研究的实验设计具有重要参考价值。

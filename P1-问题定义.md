## 一、Background（研究背景）

### 1.1 场景描述

**核心场景：企业级 Text-to-SQL 应用**

在企业级 Text-to-SQL 应用中，业务人员的提问往往包含大量的"业务术语"和"隐性逻辑"：

| 用户问法 | 数据库现状 | 缺失的中间层 |
|----------|-----------|-------------|
| "请计算上个季度高价值客户的流失率" | 只有 `t_user_log` 表，字段为 `last_login_time`, `total_spend`, `status_code` | "高价值客户"、"流失率"的定义存在于业务文档中 |

### 1.2 面临瓶颈

#### 瓶颈一：语义鸿沟（Semantic Gap）

BIRD (NeurIPS 2023) 揭示：即使是最强的 GPT-4 在真实世界数据库上仅达 54.89% 准确率（人类 92.96%）。Sequeda et al. (2023) 进一步证明：在涉及 >5 表的复杂 Schema 时，GPT-4 准确率降至 **0%**。

根本原因：用户使用业务术语提问，Schema 使用技术术语，两者无法直接对齐。

#### 瓶颈二：知识割裂（Knowledge Fragmentation）

现有 RAG 方案将"文档检索"和"Schema 检索"分开处理，无法建立业务术语与物理字段的连接。

### 1.3 已有方案与不足

**OBDA 范式（20年历史）**：
- ATHENA (Saha et al., VLDB 2016) 最早提出 Ontology-driven NL→SQL 两阶段架构
- Ontop, R2RML 等 OBDA 系统实现了 SPARQL→SQL 翻译
- **局限**：① 完全依赖手工 Ontology（构建成本高） ② 传统 NLP 无法处理复杂自然语言 ③ 全量 Schema 无筛选机制

**LLM-based Text-to-SQL**：
- RSL-SQL (2024), SchemaGraphSQL (2025) 等 SOTA 方法聚焦 Schema Linking
- **局限**：未引入结构化业务知识层，无法桥接业务术语

**核心 Gap**：OBDA 的 Ontology 思想未迁移到 LLM-based Text-to-SQL 时代。现有 LLM 方法缺乏结构化知识层，而 OBDA 方法缺乏 LLM 的语义理解能力和自动化构建能力。

## 二、Research Questions

### 主问题

**如何将 OBDA 的 Ontology 思想迁移到 LLM-based Text-to-SQL，设计低成本的 Ontology-Schema 联合召回框架？**

### 子问题分解

```
RQ1: 如何低成本构建 Ontology？
    ├─ 如何利用 LLM 从 Schema + 业务文档半自动生成 Ontology？
    ├─ Ontology 质量如何评估？不完整时如何降级？
    └─ 构建 Ontology 的人力成本是多少？

RQ2: 如何联合召回 Schema 和业务知识？
    ├─ 如何在 Ontology-Schema 两层图谱中联合召回？
    ├─ 如何提取"最小完备集"（vs OBDA 的全量查询）？
    └─ 元路径约束如何保证召回的语义正确性？

RQ3: 效果如何？
    ├─ 联合召回 vs 全量 Schema 的准确率和效率对比？
    ├─ 与 LLM-based SOTA (RSL-SQL, CHESS) 的对比？
    └─ Ontology 不完整时的鲁棒性如何？
```

> **注**：原 RQ3（评估框架）太弱，已替换为具体的效率/效果研究问题。

## 三、研究目标

### 3.1 核心目标

构建 **LLM 时代的 Ontology-Augmented Schema Linking** 框架，实现：
- "查词即查表，查表即查逻辑"
- Ontology 构建成本显著低于 OBDA 手工方案
- 在 BIRD 等标准 Benchmark 上可复现地评估

### 3.2 形式化定义

**输入**：
- 自然语言查询 $Q$
- 数据库 Schema $\mathcal{S} = \{T_1, ..., T_n\}$
- 业务 Ontology $\mathcal{O}$（半自动构建）

**输出**：
- 最小完备集 $M^* = \{T^*, C^*, K^*\}$

**完备性定义**（修订）：
$$\text{Complete}(M, Q) \iff \forall t \in \text{SQL}_{gt}(Q).\text{tables}: t \in M.T^* \land \forall c \in \text{SQL}_{gt}(Q).\text{columns}: c \in M.C^*$$

其中 $\text{SQL}_{gt}(Q)$ 是 $Q$ 对应的 ground truth SQL。完备集 $M$ 包含生成正确 SQL 所需的所有表和列。

**最小性定义**（修订）：
$$M^* = \arg\min_{M \subseteq \mathcal{S} \times \mathcal{O}} |M| \quad \text{s.t.} \quad \text{Complete}(M, Q)$$

**计算复杂度**：最小完备集问题是 Set Cover 的变种，属于 NP-hard。本研究设计贪心近似算法，近似比 $O(\log n)$。

### 3.3 量化目标

| 指标 | 基线 | 目标 | 说明 |
|------|------|------|------|
| **EX (Execution Accuracy)** | BIRD Baseline (GPT-4) | 提升 5%+ | 主指标，公平对比 |
| **Schema Recall** | RSL-SQL (94%) | 持平或提升 | 召回完整性 |
| **Token 效率** | Full Schema Prompt | 降低 30%+ | 输入 Token 数 |
| **Ontology 构建时间** | ATHENA 手工（数周） | ≤ 2 小时/数据库 | LLM 半自动 |

## 四、预期贡献

| # | 贡献 | 与现有工作的区别 |
|---|------|----------------|
| 1 | **LLM 半自动 Ontology 构建方法** | vs ATHENA 手工 OWL、OBDA 手工 R2RML |
| 2 | **Ontology-Schema 联合召回 + 最小完备集算法** | vs ATHENA 全量查询、Sequeda 全量 KG |

## 五、研究范围

### 包含
- ✅ LLM-based Text-to-SQL 中的 Schema Linking
- ✅ Ontology 半自动构建（从 Schema + 业务文档）
- ✅ Ontology-Schema 两层图谱联合召回
- ✅ BIRD 数据集上的评估

### 不包含
- ❌ 通用知识图谱构建（聚焦于数据库领域）
- ❌ Schema-free 的 Text-to-SQL
- ❌ OBDA 系统本身的优化（Ontop, R2RML）
- ❌ 完全自动化的 Ontology 构建（需要人工校验）

## 六、与先驱工作的定位

| | ATHENA (VLDB 2016) | Sequeda (2023) | 本研究 |
|--|-------------------|----------------|--------|
| **时代** | 传统 NLP | LLM (GPT-4) | LLM 时代 |
| **方法** | 规则+同义词 | Zero-shot Prompt | 联合召回算法 |
| **Ontology** | 手工 OWL2 | 手工 OWL+R2RML | **LLM 半自动** |
| **查询** | 全量 Schema | 全量 KG | **最小完备集** |
| **推理** | 排序 | 无 | **元路径约束** |
| **评估** | 自定义 | 自定义 | **BIRD 标准** |

**定位**：本研究不是"首次引入 Ontology"，而是"将 OBDA 思想带入 LLM 时代的 Schema Linking，并解决构建成本、召回效率、语义约束三个遗留问题"。

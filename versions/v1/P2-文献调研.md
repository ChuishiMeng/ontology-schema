## 1. Text-to-SQL 研究综述

### 1.1 发展历程

Text-to-SQL 的研究可追溯到 1973 年的 LUNAR 系统，用于回答月球岩石相关问题。发展经历了三个主要阶段：

| 阶段 | 时间 | 代表方法 | 特点 |
|------|------|----------|------|
| **规则时代** | 1973-2015 | LUNAR, Grammar-based | 依赖人工规则，领域受限 |
| **神经网络时代** | 2015-2022 | Seq2Seq, Transformer | 数据驱动，泛化能力提升 |
| **LLM 时代** | 2022-至今 | GPT-4, DIN-SQL, DAIL-SQL | 零样本/少样本，能力飞跃 |

### 1.2 核心挑战（文献总结）

根据 Mohammadjafari et al. (2024) 的综述，Text-to-SQL 面临四大挑战：

#### C1: 语言学复杂性和歧义
- 自然语言包含嵌套从句、代词、模糊术语
- 例：查询中 "区域" 可能指向未明确提及的特定地理位置

#### C2: Schema 理解与表示
- 需要深度理解数据库 Schema（表、列、关系、约束）
- 实际应用中 Schema 复杂且跨域差异大
- 正确映射自然语言到 Schema 是核心难点

#### C3: 罕见和复杂 SQL 操作
- 嵌套子查询、多表 JOIN、窗口函数
- 训练数据中不常见的操作难以生成

#### C4: 跨域泛化
- 在特定领域（如客服数据库）训练的模型难以泛化到其他领域（如医疗）
- Schema 结构、术语、查询模式差异导致性能下降

### 1.3 LLM-based Text-to-SQL 方法分类

根据 Survey (2024) 的分类，LLM-based 方法分为四类：

```
LLM-based Text-to-SQL
├── Prompt Engineering（无训练）
│   ├── Zero-Shot: 直接生成 SQL
│   ├── Few-Shot: 示例驱动的生成
│   └── Chain-of-Thought: 分步推理
│
├── Fine-Tuning（从预训练模型微调）
│   ├── Full-Parameter: 全参数微调
│   └── PEFT: 参数高效微调（LoRA等）
│
├── Task-Training（从零训练）
│   ├── Transformer-based
│   └── Mixture of Experts
│
└── LLM Agent（多智能体+外部工具）
    ├── 多智能体协作
    ├── 动态 SQL 生成与修正
    └── 外部工具集成
```

### 1.4 代表性方法

| 方法 | 类型 | 核心思想 | Spider EX | BIRD EX |
|------|------|----------|-----------|---------|
| **DIN-SQL** | Prompt | 分解式推理 + Schema 链接 | 85.3% | - |
| **DAIL-SQL** | Prompt | 自我一致性 + 执行引导 | 86.6% | - |
| **CHASE-SQL** | Prompt | Gemini + 高级推理 | - | 72.28% |
| **C3** | Prompt | 零样本 + ChatGPT | 82.3% | - |
| **LinkAlign** | Agent | 多轮语义检索 + Schema 提取 | - | SOTA |

## 2. Schema Linking 研究

### 2.1 问题定义

**Schema Linking**：识别用户查询所需的数据库 Schema（表和列），是 Text-to-SQL 的关键瓶颈。

根据 Wang & Liu (2025) 的 LinkAlign 论文，Schema Linking 面临两大挑战：

```
Challenge 1: Database Retrieval
├── 多数据库环境下的正确数据库选择
└── 过滤无关数据库

Challenge 2: Schema Item Grounding
├── 从大型冗余 Schema 中识别相关表和列
└── 语义相似的表/列增加遗漏风险
```

### 2.2 传统方法

| 方法 | 技术 | 局限性 |
|------|------|--------|
| **精确匹配** | 字符串匹配 | 无法处理语义等价 |
| **TF-IDF** | 关键词相似度 | 忽略上下文语义 |
| **编码器方法** | BERT 嵌入 | 缺乏结构信息 |

### 2.3 现代方法

#### LinkAlign (2025)
- **多轮语义增强检索**：通过查询重写进行语义对齐
- **无关信息隔离**：过滤无关 Schema 元素
- **Schema 提取增强**：精确识别相关表和列
- **性能**：Spider 2.0-lite 上达到 SOTA（排除长 CoT 模型）

#### TSL (2025)
- **表示层**：Transformer 联合学习表和列嵌入
- **密集分类层**：MLP 投影到链接预测
- **Schema 注解**：Token 级分类任务
- **鲁棒性**：对抗条件下保持语义对齐

### 2.4 Schema Linking 的研究空白

| 现有方法局限 | 具体表现 |
|--------------|----------|
| **缺乏业务知识** | 仅依赖 Schema 语义，忽略业务规则 |
| **静态表示** | 无法处理 Schema 演化 |
| **单一信号** | 未融合多种召回信号 |
| **规模限制** | 大型 Schema (>1000 列) 性能下降 |

## 3. 知识图谱与 Ontology 在数据系统中的应用

### 3.1 Ontology 与 Knowledge Graph 的关系

根据文献综述：

| 概念 | 定义 | 作用 |
|------|------|------|
| **Ontology** | 通用语义数据模型 | 定义实体类型、关系、约束 |
| **Knowledge Graph** | Ontology + 实例数据 | 存储具体实体和关系 |

> "Ontologies are generalized semantic data models, while a knowledge graph is what we get when we leverage that model and apply it to instance data." — Enterprise Knowledge

### 3.2 Ontology-Driven Knowledge Graph for GraphRAG

deepsense.ai (2025) 提出的方法：

```
Ontology-Driven KG Construction
├── 定义 RDF Ontology（实体类型、关系）
├── LLM + Ontology 约束 → 实体抽取
├── Neo4j 图数据库存储
└── 向量检索 + 图遍历 → RAG
```

**核心价值**：
- **Grounding LLM**：约束 LLM 生成符合预期的实体和关系
- **语义形式化**：RDF 表示提供更精确的语义
- **可追溯性**：知识来源可追溯

### 3.3 Virtual Knowledge Graph

Reddit 社区讨论的方案：

```
Relational Database → Ontology Mapping → Virtual KG → SPARQL/GraphQL
```

**关键技术**：
- **R2RML 映射**：关系数据库到 RDF 的映射
- **语义层抽象**：隐藏 JOIN 逻辑
- **统一查询接口**：SPARQL/GraphQL

### 3.4 Ontology 在数据库查询中的应用

| 应用场景 | Ontology 作用 | 示例 |
|----------|---------------|------|
| **语义查询** | 理解业务术语 | "高价值客户" → 定义规则 |
| **Schema 增强** | 为字段添加语义标注 | customer.status → 生命周期阶段 |
| **推理支持** | 推断隐含关系 | A 购买 B，B 属于 C → A 间接关联 C |
| **歧义消除** | 提供上下文约束 | "去年" → fiscal year 或 calendar year |

### 4.2 Spider 2.0：真实世界挑战

Spider 2.0 是当前最具挑战性的 Text-to-SQL 基准：

```
Spider 2.0 特点
├── 复杂数据环境：>3000 列的大型 Schema
├── 多 SQL 方言：BigQuery, Snowflake, PostgreSQL
├── 多步骤操作：ETL、转换、查询
├── 真实企业场景：Cloudflare, Google等真实数据库
└── 文档依赖：需要阅读数据库文档
```

**Leaderboard (2026-02)**:
| 排名 | 方法 | 分数 |
|------|------|------|
| 1 | QUVI-3 + Gemini-3-pro | 94.15 |
| 2 | TCDataAgent-SQL | 93.97 |
| 3 | Native mini | 92.50 |
| - | Ask Data + Relational KG | 86.28 |
| - | Spider-Agent + Claude-4-Sonnet | 25.78 |
| - | DIN-SQL + GPT-4o | 0.00 |

**观察**：传统 Text-to-SQL 方法在真实场景几乎完全失效，凸显 Schema Linking 的重要性。

### 4.3 评估指标

| 指标 | 定义 | 适用场景 |
|------|------|----------|
| **Exact Match (EM)** | SQL 完全匹配 | Spider |
| **Execution Accuracy (EX)** | 执行结果正确 | Spider, BIRD |
| **Valid Efficiency Score (VES)** | 效率评分 | BIRD |
| **Test-Suite Accuracy** | 多测试用例通过率 | 鲁棒性评估 |

## 5. 相关工作差距分析

### 5.1 现有方法的不足

| 方法类别 | 不足 | 影响 |
|----------|------|------|
| **Schema-Only** | 缺乏业务语义理解 | 无法处理业务术语 |
| **LLM-Only** | 无结构化知识约束 | 幻觉、错误 Schema |
| **Ontology-Only** | 无具体数据映射 | 无法生成可执行 SQL |
| **传统 Schema Linking** | 单一召回信号 | 召回不完整 |

### 5.2 研究机会

```
Ontology-Schema 联合召回的研究空白
├── Gap 1: 缺乏 Ontology-Schema 统一表示框架
│   └── 机会：设计多模态知识嵌入方法
│
├── Gap 2: 现有 Schema Linking 忽略业务知识
│   └── 机会：融合语义、结构、规则三种信号
│
├── Gap 3: LLM 生成 SQL 缺乏知识约束
│   └── 机会：知识引导的 SQL 生成
│
└── Gap 4: 端到端评估缺失
    └── 机会：构建完整评测框架
```

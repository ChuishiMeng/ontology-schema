## 1. 现有方法系统性不足

### 1.1 表示层不足

#### 1.1.1 Ontology 表示的局限

| 局限 | 具体表现 | 影响 |
|------|----------|------|
| **静态性** | Ontology 构建后难以自动更新 | 业务变化时知识过时 |
| **粒度不匹配** | 概念抽象程度与 Schema 字段不一致 | 对齐困难 |
| **LLM 不友好** | RDF/OWL 格式不适合 LLM 理解 | Prompt 冗长 |
| **缺乏上下文** | 孤立的实体定义，无使用场景 | 难以应用于具体查询 |

**实证**：企业 Ontology 通常需要专人维护，更新周期 > 1 个月。

#### 1.1.2 Schema 表示的局限

| 局限 | 具体表现 | 影响 |
|------|----------|------|
| **语义缺失** | DDL 只有结构，无业务含义 | "status" 字段含义不明 |
| **命名不规范** | 缩写、非标准命名 | "cust_flg" 难以理解 |
| **规模膨胀** | 企业级 >3000 列 | Token 限制 |
| **版本混乱** | 多版本 Schema 共存 | 召回歧义 |

**实证**：Spider 2.0 显示传统 Text-to-SQL 方法在大规模 Schema 下准确率接近 0%。

#### 1.1.3 联合表示的缺失

```
现有方法                     理想状态
─────────────────           ─────────────────
Ontology (独立)              Ontology ─┐
      ↓                            ↓  ├→ 联合表示空间
Schema (独立)                Schema ───┘
      ↓
简单拼接/拼接
```

**核心问题**：没有统一的表示框架同时编码 Ontology 和 Schema。

### 1.2 召回层不足

#### 1.2.1 Schema Linking 的局限

根据 P2 文献调研，现有 Schema Linking 方法存在以下问题：

| 方法 | 信号源 | 召回率 | 局限性 |
|------|--------|--------|--------|
| **精确匹配** | 字符串 | ~60% | 无法处理同义词 |
| **语义相似度** | 向量 | ~75% | 忽略结构关系 |
| **GNN 编码** | 图结构 | ~80% | 无业务规则 |
| **LLM 推理** | 语义 | ~85% | Token 限制、幻觉 |

**实证**：
- LinkAlign (2025) 在 Spider 2.0 上最高 94.15%，但依赖长 CoT 推理
- DIN-SQL + GPT-4o 在 Spider 2.0 上 0% 准确率

#### 1.2.2 缺乏业务知识召回

```
现有 Schema Linking:
Query → Schema Retrieval → SQL
         (仅 Schema)

理想联合召回:
Query → Schema Retrieval + Ontology Retrieval + Rule Retrieval → SQL
         (多源知识)
```

**缺失的业务知识类型**：
1. **业务术语定义**："高价值客户" 的具体标准
2. **计算规则**："退货率" = Returns / Total
3. **业务约束**：查询时间窗口、数据权限
4. **隐含关系**：业务概念间的关系（非 FK）

#### 1.2.3 召回信号融合不足

| 融合方式 | 现有方法 | 问题 |
|----------|----------|------|
| **Late Fusion** | 各召回器独立，最后合并 | 信号冲突 |
| **Weighted Sum** | 固定权重融合 | 不自适应 |
| **无融合** | 单一召回器 | 召回不完整 |

### 1.3 生成层不足

#### 1.3.1 LLM 幻觉问题

```
场景: 查询 "客户购买趋势"
Schema: customers, orders, order_items

LLM 幻觉示例:
SELECT customer_trend_score  -- 不存在的字段
FROM customer_analytics       -- 不存在的表
WHERE analysis_type = 'purchase'  -- 不存在的过滤条件
```

**原因**：缺乏 Schema 约束，LLM 自由生成不存在元素。

#### 1.3.2 缺乏业务规则约束

```
场景: 查询 "退货率最高的产品"
业务规则: 退货率 = 退货数量 / 总销售数量 * 100%

LLM 生成（错误）:
SELECT product_id, 
       COUNT(return_id) as return_rate  -- 错误的计算
FROM returns
GROUP BY product_id
ORDER BY return_rate DESC

正确（需要业务知识）:
SELECT p.product_id,
       COUNT(DISTINCT r.order_id) * 100.0 / 
       COUNT(DISTINCT o.order_id) as return_rate
FROM products p
LEFT JOIN orders o ON p.product_id = o.product_id
LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY p.product_id
ORDER BY return_rate DESC
```

#### 1.3.3 复杂查询生成能力不足

| 查询类型 | 现有方法准确率 | 根本原因 |
|----------|----------------|----------|
| **单表简单** | ~90% | 模式简单 |
| **2表 JOIN** | ~75% | 需要识别 FK |
| **3+表 JOIN** | ~50% | 路径推理难 |
| **嵌套子查询** | ~30% | 逻辑复杂 |
| **窗口函数** | ~20% | 训练数据少 |

### 1.4 评估层不足

#### 1.4.1 评估指标不完整

| 现有指标 | 覆盖范围 | 缺失部分 |
|----------|----------|----------|
| **Exact Match** | SQL 语法 | 语义等价但语法不同 |
| **Execution Accuracy** | 执行结果 | 召回过程评估 |
| **VEM** | 效率 | 业务正确性 |

#### 1.4.2 缺乏组件级评估

```
现有评估: 端到端 (Query → SQL)
         ↓
    单一指标

理想评估: 分组件
    Query → [Recall] → [Generation] → SQL
              ↓            ↓
         Recall@K    Generation Accuracy
```

#### 1.4.3 缺乏业务场景评估

| 评估维度 | 现有覆盖 | 缺失场景 |
|----------|----------|----------|
| **领域** | 通用 | 特定行业（金融、医疗） |
| **规模** | 小型 | 企业级（>1000 列） |
| **复杂度** | 学术 | 生产级（脏数据、演化） |

## 3. 差异化创新点

### 3.1 核心创新：Ontology-Schema 联合召回框架

#### 3.1.1 创新定义

```
OS-Joint: Ontology-Schema Joint Retrieval Framework

输入:
- 自然语言查询 Q
- 数据库 Schema S
- 业务 Ontology O

输出:
- 相关表集合 T* ⊆ S
- 相关列集合 C*
- 相关业务知识 K* ⊆ O

创新:
1. 统一表示: JointEmbedding(O, S) → V
2. 多源召回: Semantic + Structural + Rule-based
3. 融合排序: Learned Fusion + Re-ranking
```

#### 3.1.2 与现有方法的差异

| 维度 | 现有方法 | OS-Joint |
|------|----------|----------|
| **知识源** | Schema Only | Schema + Ontology |
| **表示** | 独立编码 | 联合嵌入 |
| **召回** | 单一信号 | 多源融合 |
| **排序** | 相似度排序 | 学习排序 |
| **约束** | 无 | 知识约束 |

### 3.2 具体创新点

#### 创新点 1：统一知识表示 (Unified Knowledge Representation)

**问题**：Ontology 和 Schema 在不同空间表示。

**创新**：
```python
class UnifiedKnowledgeRepresentation:
    """
    联合表示 Ontology 和 Schema 的统一框架
    """
    def __init__(self, ontology, schema):
        # 1. 实体对齐
        self.alignment = self.align_entities(ontology, schema)
        
        # 2. 图构建
        self.unified_graph = self.build_unified_graph(ontology, schema, self.alignment)
        
        # 3. 联合嵌入
        self.embeddings = self.joint_embedding(self.unified_graph)
    
    def joint_embedding(self, graph):
        """
        使用混合 GNN 编码统一知识图
        """
        # 节点类型: Ontology_Entity, Schema_Table, Schema_Column
        # 边类型: ont_relation, fk_relation, align_relation
        
        embeddings = HeteroGNN(
            node_types=['ont_entity', 'schema_table', 'schema_column'],
            edge_types=['ont_rel', 'fk_rel', 'align_rel', 'has_attr']
        )(graph)
        
        return embeddings
```

**优势**：
- 语义等价元素在向量空间接近
- 保留结构信息
- 支持跨模态检索

#### 创新点 2：多源融合召回 (Multi-Source Fusion Retrieval)

**问题**：单一召回信号不完整。

**创新**：
```python
class MultiSourceFusionRetrieval:
    """
    融合语义、结构、规则三种召回信号
    """
    def recall(self, query, unified_rep):
        # 1. 语义召回
        sem_scores = self.semantic_recall(query, unified_rep.embeddings)
        
        # 2. 结构召回
        struct_scores = self.structural_recall(query, unified_rep.unified_graph)
        
        # 3. 规则召回
        rule_scores = self.rule_recall(query, unified_rep.ontology.rules)
        
        # 4. 学习融合
        fused_scores = self.learned_fusion(sem_scores, struct_scores, rule_scores)
        
        # 5. 重排序
        ranked_results = self.rerank(query, fused_scores)
        
        return ranked_results
    
    def learned_fusion(self, sem, struct, rule):
        """
        使用学习到的权重融合多源分数
        """
        # 自适应权重: 根据查询类型调整
        weights = self.weight_predictor([sem, struct, rule])
        fused = weights[0] * sem + weights[1] * struct + weights[2] * rule
        return fused
```

**优势**：
- 互补信息融合
- 自适应权重
- 召回完整

#### 创新点 3：知识约束 SQL 生成 (Knowledge-Constrained SQL Generation)

**问题**：LLM 生成存在幻觉、不符合业务规则。

**创新**：
```python
class KnowledgeConstrainedSQLGeneration:
    """
    使用召回的知识约束 SQL 生成
    """
    def generate(self, query, recalled_knowledge):
        prompt = self.build_constrained_prompt(query, recalled_knowledge)
        sql = self.llm.generate(prompt)
        sql = self.validate_and_correct(sql, recalled_knowledge)
        return sql
    
    def build_constrained_prompt(self, query, knowledge):
        return f"""
## Available Tables (USE ONLY THESE)
{knowledge.tables}

## Available Columns (USE ONLY THESE)
{knowledge.columns}

## Business Rules
{knowledge.rules}

## Query
{query}

## Constraints
1. Use ONLY the tables and columns listed above
2. Follow the business rules
3. If you need information not in the available tables, respond "CANNOT ANSWER"

Generate SQL:
"""
    
    def validate_and_correct(self, sql, knowledge):
        """
        验证 SQL 只使用召回的元素
        """
        used_tables = extract_tables(sql)
        used_columns = extract_columns(sql)
        
        if not set(used_tables).issubset(set(knowledge.tables)):
            sql = self.correct_tables(sql, knowledge.tables)
        
        if not set(used_columns).issubset(set(knowledge.columns)):
            sql = self.correct_columns(sql, knowledge.columns)
        
        return sql
```

**优势**：
- 消除幻觉
- 业务规则遵守
- 可解释性

#### 创新点 4：端到端评估框架 (End-to-End Evaluation Framework)

**问题**：缺乏组件级和业务评估。

**创新**：
```python
class EvaluationFramework:
    """
    多层次评估框架
    """
    def evaluate(self, system, test_set):
        results = {}
        
        # 1. 组件级评估
        results['recall'] = self.evaluate_recall(system, test_set)
        results['generation'] = self.evaluate_generation(system, test_set)
        
        # 2. 端到端评估
        results['end2end'] = self.evaluate_end2end(system, test_set)
        
        # 3. 业务评估
        results['business'] = self.evaluate_business(system, test_set)
        
        return results
    
    def evaluate_recall(self, system, test_set):
        """
        召回组件评估
        """
        recall_at_k = {1: [], 5: [], 10: []}
        
        for sample in test_set:
            recalled = system.recall(sample.query)
            for k in [1, 5, 10]:
                hit = any(t in recalled[:k] for t in sample.ground_truth_tables)
                recall_at_k[k].append(hit)
        
        return {f"Recall@{k}": np.mean(v) for k, v in recall_at_k.items()}
    
    def evaluate_business(self, system, test_set):
        """
        业务场景评估
        """
        metrics = {
            'rule_compliance': [],  # 业务规则遵守率
            'executability': [],    # SQL 可执行率
            'correctness': []       # 业务正确性（人工评估）
        }
        # ...
        return metrics
```

**优势**：
- 细粒度诊断
- 业务对齐
- 可追溯

## 5. 风险与缓解

### 5.1 技术风险

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| **对齐质量** | Ontology-Schema 对齐不准确 | 人工验证 + 主动学习 |
| **召回延迟** | 多源召回延迟高 | 并行召回 + 缓存 |
| **LLM 不可控** | 约束 Prompt 仍可能违反 | 后处理验证 |
| **Ontology 维护** | 需要持续更新 | 自动推断 + 增量更新 |

### 5.2 数据风险

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| **标注成本** | 人工标注对齐关系昂贵 | 半自动标注 |
| **领域差异** | 特定领域数据不足 | 迁移学习 |
| **Schema 变化** | 企业 Schema 演化 | 版本管理 |

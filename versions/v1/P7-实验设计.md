## 2. Baseline 方法

### 2.1 DIN-SQL

| 属性 | 说明 |
|------|------|
| **来源** | Text-to-SQL SOTA (EM 2023) |
| **核心思想** | 依赖关系驱动的中间表示 |
| **特点** | 显式建模表间依赖关系 |
| **召回策略** | 基于依赖图的表选择 |

**论文**: "DIN-SQL: Decomposed In-Context Learning of Text-to-SQL"

### 2.2 RAT-SQL

| 属性 | 说明 |
|------|------|
| **来源** | ICLR 2021 |
| **核心思想** | 关系感知Transformer |
| **特点** | 显式编码表/列关系 |
| **召回策略** | Schema Encoding + Linking |

**论文**: "RAT-SQL: Relation-Aware Transformer for Text-to-SQL"

### 2.3 Text-to-SQL (PICARD)

| 属性 | 说明 |
|------|------|
| **来源** | arXiv 2021 |
| **核心思想** | 增量式解码 + 约束验证 |
| **特点** | 逐token生成，保证语法正确 |
| **召回策略** | 宽松召回 + 生成时验证 |

**论文**: "PICARD: Parsing Incrementally for Constrained Auto-Regressive Decoding"

### 2.4 简化 Baseline

| 方法 | 说明 |
|------|------|
| **Random** | 随机召回表/列 |
| **BM25** | 传统关键词检索 |
| **SBERT** | 纯语义向量检索 |

## 3. 主实验设计

### 3.1 实验设置

```
实验环境:
- GPU: NVIDIA A100 x 1
- Python: 3.9+
- 框架: PyTorch 2.0, transformers 4.30
- LLM: GPT-4 (API调用), CodeGen-16B (本地)
- 向量模型: bge-large-zh-v1.5 (中文), bge-large-en-v1.5 (英文)
```

### 3.2 对比实验

| 方法 | 召回策略 | 生成策略 | 说明 |
|------|----------|----------|------|
| **Ours (Full)** | Ontology+Schema+Rule | LLM + 验证 | 完整方案 |
| **DIN-SQL** | 依赖图 | In-context Learning | SOTA Baseline |
| **RAT-SQL** | Relation-aware | Transformer | SOTA Baseline |
| **PICARD** | 宽松召回 | Constrained Decoding | SOTA Baseline |
| **SBERT** | 纯语义 | LLM | 无结构/规则 |

### 3.3 评估指标

| 层级 | 指标 |
|------|------|
| **召回层** | Recall@1/3/5, MRR, NDCG@5 |
| **生成层** | Valid Rate, Exec Acc, Exact Match |
| **端到端** | Task Success Rate, Latency |

### 3.4 假设检验

使用配对t检验验证方法间差异显著性 (α=0.05)

| **-Ontology** | Ontology编码+召回 | Schema召回+Rule召回 |

## 6. Case Study 设计

### 6.1 案例选择标准

| 标准 | 说明 |
|------|------|
| **典型性** | 代表某一类问题 |
| **挑战性** | 包含难点/坑点 |
| **可解释性** | 能清晰展示方法优势 |
| **多样性** | 覆盖不同查询类型 |

### 6.2 案例列表

#### Case 1: 多表JOIN + 业务概念

```
Query: "找出2024年第一季度销售额超过100万的活跃客户"

难点:
- 时间范围转换 (Q1 2024 → 01-01~03-31)
- 业务概念理解 (活跃=has_order_recently)
- 多表JOIN (customers → orders → order_items)

对比分析:
- 纯Schema: 召回customers, orders, order_items
- Ours: 额外召回 business_concept:活跃, time_rule:Q1
```

#### Case 2: 复杂聚合 + 隐含关系

```
Query: "计算每个部门的离职率"

难点:
- 隐含计算 (离职率 = 离职人数 / 总人数)
- 列不在表面Schema (需要派生)
- 部门层级理解

对比分析:
- 纯Schema: 无法理解"离职率"计算方式
- Ours: 召回 rule:离职率=SUM(left)/COUNT(*) + related_columns
```

#### Case 3: 中文领域术语

```
Query: "查询最近一周内下单的客户购买力分布"

难点:
- 中文领域术语 (购买力, 分布)
- 时间相对表达 (最近一周)
- 分布查询 (需要分桶)

对比分析:
- 英文模型: 难以理解中文术语
- Ours: 召回 ontology:购买力 关联到 spending_level
```

### 6.3 分析维度

| 维度 | 指标 |
|------|------|
| **召回质量** | 召回表/列是否正确完整 |
| **知识利用** | 是否正确使用Ontology/Rule |
| **生成质量** | SQL语法、执行结果 |
| **失败分析** | 失败原因分类 |

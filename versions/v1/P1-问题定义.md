## 一、Background（研究背景）

### 1.1 场景描述

**核心场景：企业级 Text-to-SQL 应用**

在企业级 Text-to-SQL 应用中，业务人员的提问往往包含大量的"业务术语"和"隐性逻辑"：

| 用户问法 | 数据库现状 | 缺失的中间层 |
|----------|-----------|-------------|
| "请计算上个季度高价值客户的流失率" | 只有 `t_user_log` 表，字段为 `last_login_time`, `total_spend`, `status_code` | "高价值客户"、"流失率"的定义存在于业务文档中（如：高价值=消费>1000元，流失=30天未登录） |

**关键特征**：
- 用户使用业务黑话（Domain Jargon），而非数据库术语
- 业务逻辑隐含在文档中，而非数据库 Schema 中
- 查询需要多跳推理：概念 → 公式 → 字段

### 1.2 面临瓶颈

#### 瓶颈一：语义鸿沟（Semantic Gap）

传统的 Schema Linking 仅关注"问题词汇"与"表列名"的字符或向量匹配。当问题涉及必须经由外部知识推导的逻辑时，传统方法会失效。

**失效案例**：
```
用户查询: "找出去年高价值客户的购买趋势"

Schema-Only 方法:
1. 找到 products 表的 category 字段
2. 找到 orders/returns 表
3. 执行 JOIN 和聚合

❌ 失效原因:
- 不知道"高价值客户"如何定义（需要业务规则）
- 不知道"购买趋势"如何计算（需要业务概念）
- 不知道"去年"的精确含义（fiscal year vs calendar year）
```

#### 瓶颈二：知识割裂（Knowledge Fragmentation）

现有的 RAG 方案通常将"文档检索"和"Schema 检索"分开处理：

| 问题 | 表现 |
|------|------|
| **召回了正确的 Schema，但不知道计算公式** | LLM 知道用 `orders` 表，但不知道"流失率"怎么算 |
| **召回了公式，但找不到对应的列** | LLM 知道"消费>1000元"，但不知道对应哪个字段 |
| **无法建立概念与物理层的连接** | 业务术语与数据库列名之间存在巨大鸿沟 |

### 1.3 发展现状

目前的 SOTA 正在从单纯的 **Text-to-SQL** 向 **Knowledge-Augmented Text-to-SQL** 演进：

| 方法类别 | 代表作 | 核心思路 | 局限性 |
|----------|--------|---------|--------|
| **Embedding-based Schema Linking** | RAT-SQL, RESDSQL | 向量相似度匹配 | 仅基于字面/语义匹配，缺乏业务逻辑理解 |
| **Graph-based Schema Linking** | BOGIN | GNN 在 Schema 图上传播 | 仅限 DB 内部关系，未包含外部业务知识 |
| **RAG-based Knowledge Retrieval** | BIRD-Baseline, MAC-SQL | 文本块检索 | 知识是非结构化的，难以对齐到具体列 |

**研究趋势**：从"如何更好地编码 Schema"转向"如何构建一个包含 Schema 和 Domain Knowledge 的统一视图"。

## 二、What（研究目标）

### 2.1 Goal（定位与理念）

**研究定位**：构建一个异构图谱驱动的联合召回系统（Heterogeneous Graph-based Joint Retrieval System）

**核心理念**：
- 将"数据库结构（Schema）"和"业务逻辑（Knowledge）"视为同一语义空间中的实体
- 通过图谱（Ontology）将二者显式连接
- 实现"查词即查表，查表即查逻辑"

**科学定义**：

给定自然语言查询 $Q$，在一个包含 Schema 节点 $S$ 和知识节点 $K$ 的异构图谱 $G=(V, E)$ 中，联合检索出一个最优子图 $G'$，使得 $G'$ 包含生成正确 SQL 所需的**最小完备集（Minimal Context Set）**。

$$
\text{JointRetrieval}(Q, G) \rightarrow G' = \{T^*, C^*, K^*\}
$$

其中：
- $T^*$：相关的表集合
- $C^*$：相关的字段/列集合
- $K^*$：相关的业务知识（概念、公式、规则）

### 2.2 Results（预期成果）

#### 量化指标

| 指标 | 定义 | 目标 | 对比基线 |
|------|------|------|----------|
| **Recall@K** | Schema 召回率与知识 Snippet 召回率的加权平均值 | 提升 10%+ | RESDSQL, BOGIN |
| **EX (Execution Accuracy)** | SQL 执行准确率（BIRD 数据集） | 提升 5%+ | BIRD Baseline |
| **Token Efficiency** | LLM 输入 Token 消耗降低 | 降低 30%+ | Full Schema Prompt |

#### 关键交付物

1. **三层异构图谱构建方法**：L0（元模型）→ L1（业务层）→ L2（物理层）
2. **联合召回算法**：元路径约束下的图游走与子图提取
3. **端到端系统原型**：支持"用户 Query → 业务概念 → 计算公式 → 物理字段"的可视化推理路径
4. **评估基准**：在 BIRD 等数据集上的完整实验

## 三、Why（研究意义）

### 3.1 Values（价值）

| 价值维度 | 具体内容 |
|----------|---------|
| **客户价值** | 解决"幻觉"问题，让非技术人员使用业务黑话查询数据，真正实现数据民主化 |
| **效率价值** | 通过联合召回修剪无关 Schema 和文档，降低 LLM 推理的上下文长度和延迟 |
| **维护价值** | 将业务逻辑从 Prompt 工程中解耦，沉淀为知识图谱，形成企业的"指标字典" |
| **学术价值** | 首次系统性地将非结构化业务文档转化为结构化 Ontology，并与 Schema 融合 |

### 3.2 Challenges（挑战）

| 挑战 | 具体问题 | 本研究应对思路 |
|------|---------|---------------|
| **异构性挑战** | 非结构化文本（如"净利润等于收入减成本"）如何与结构化图（Table: Finance）对齐？ | 三层图谱架构 + LLM 驱动的实例化 |
| **噪声干扰** | 纯关键词匹配（BM25）容易召回大量无关文档 | 元路径约束推理，过滤不合逻辑的路径 |
| **多跳推理** | 用户问 A（概念）→ A 依赖 B（公式）→ B 对应 C（字段） | 图谱显式建模传递关系 |

## 四、研究问题（Research Questions）

### 4.1 核心问题

**主问题**：如何构建统一的语义表示空间，实现业务知识与数据库 Schema 的联合召回？

### 4.2 子问题分解

```
RQ1: 如何表示？—— 统一语义空间的构建
    ├─ 如何设计异构图谱的层次结构？
    ├─ 如何将非结构化业务文档转化为结构化 Ontology？
    └─ 如何建立 Ontology 与 Schema 的语义连接？

RQ2: 如何召回？—— 联合检索机制
    ├─ 如何识别 Query 中的业务概念和计算逻辑？
    ├─ 如何利用图谱结构进行约束推理？
    └─ 如何提取"最小完备集"子图？

RQ3: 如何验证？—— 评估框架
    ├─ 如何评估召回的完整性和准确性？
    ├─ 如何评估对 SQL 生成质量的提升？
    └─ 如何与现有 SOTA 方法进行公平对比？
```

### 4.3 问题形式化

**输入**：
- 用户自然语言查询 $Q$
- 数据库 Schema $\mathcal{S} = \{T_1, T_2, ..., T_n\}$
- 业务知识库 $\mathcal{K}$（非结构化文档 + 结构化规则）

**输出**：
- 最小完备集 $G' = \{T^*, C^*, K^*\}$

**约束**：
- $G'$ 必须包含生成正确 SQL 的所有必要信息
- $G'$ 应尽可能小，避免引入噪声
- $G'$ 的提取过程应可解释

## 五、研究范围界定

### 5.1 包含

- ✅ Text-to-SQL 中的 Schema Linking 问题
- ✅ 业务知识与数据库 Schema 的语义对齐
- ✅ 异构图谱的构建与推理
- ✅ 联合召回算法设计
- ✅ BIRD 等知识增强数据集上的评估

### 5.2 不包含

- ❌ 通用知识图谱构建（非本研究重点）
- ❌ 纯 Schema-free 的 Text-to-SQL 方法
- ❌ 非 SQL 类型的数据查询（如 API 生成）
- ❌ 实时流数据处理

## 1. Ontology 表示方法

### 1.1 传统 Ontology 表示

#### 1.1.1 RDF (Resource Description Framework)

```xml
<rdf:Description rdf:about="Customer">
  <rdf:type rdf:resource="BusinessEntity"/>
  <hasAttribute rdf:resource="customer_id"/>
  <hasAttribute rdf:resource="customer_name"/>
  <hasRelationship rdf:resource="placesOrder"/>
</rdf:Description>
```

**特点**：
- 三元组形式：(Subject, Predicate, Object)
- 标准化、可推理
- 冗长、不适合 LLM 输入

#### 1.1.2 OWL (Web Ontology Language)

```owl
Class: Customer
  SubClassOf: BusinessEntity
  hasAttribute: customer_id, customer_name

ObjectProperty: placesOrder
  Domain: Customer
  Range: Order
```

**特点**：
- 支持复杂约束（基数、传递性）
- 推理能力强
- 学习曲线陡峭

### 1.2 现代 Ontology 表示

#### 1.2.1 知识图谱嵌入

| 方法 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **TransE** | h + r ≈ t | 简单高效 | 1-N 关系差 |
| **TransH** | 超平面投影 | 处理 1-N | 复杂度增加 |
| **RotatE** | 复数旋转 | 多种关系模式 | 计算成本高 |
| **BERT-Embedding** | 上下文嵌入 | 语义丰富 | 无结构信息 |

#### 1.2.2 LLM 友好的 Ontology 表示

**JSON-LD 格式**：
```json
{
  "@context": {
    "hasAttribute": "http://example.org/ontology#hasAttribute",
    "placesOrder": "http://example.org/ontology#placesOrder"
  },
  "@type": "Customer",
  "hasAttribute": ["customer_id", "customer_name"],
  "placesOrder": {"@type": "Order"}
}
```

**自然语言描述**：
```
Customer is a business entity representing a person or organization 
that purchases products or services.

Attributes:
- customer_id: unique identifier (integer, primary key)
- customer_name: full name of the customer (string)

Relationships:
- placesOrder: Customer can place one or more Orders
- hasStatus: Customer has a lifecycle status (New/Active/Churned)

Business Rules:
- High-value customer: total_order_amount > 10000 in last 12 months
```

### 1.3 Ontology 表示的挑战

| 挑战 | 描述 | 现有解决方案 |
|------|------|--------------|
| **自动构建** | 从业务文档自动提取 | LLM + 规则模板 |
| **演化更新** | 业务变化时保持同步 | 增量更新机制 |
| **领域适应** | 跨领域迁移 | 预训练 + 微调 |
| **质量保证** | 一致性、完整性验证 | 约束检查 + 人工审核 |

## 2. Schema 表示方法

### 2.1 传统 Schema 表示

#### 2.1.1 DDL (Data Definition Language)

```sql
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    email VARCHAR(255),
    created_at TIMESTAMP
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    order_date DATE,
    total_amount DECIMAL(10,2)
);
```

**特点**：
- 完整的结构信息
- 无语义信息
- 需要专业知识理解

#### 2.1.2 Schema Graph

```
[customers] --1:N--> [orders]
     |                    |
     v                    v
[customer_id]        [order_id]
[customer_name]      [customer_id]
[email]              [order_date]
```

### 2.2 现代 Schema 表示

#### 2.2.1 Schema Embedding 方法

**GNN-based Schema Encoding**：
```
Input: Schema Graph G = (V, E)
V = {tables, columns}
E = {foreign keys, table-column relations}

Process:
1. 初始化节点嵌入 (Table/Column 名称 + 类型)
2. GNN 传播 (GraphSAGE/GAT)
3. 池化得到 Schema 表示

Output: h_table, h_column for each node
```

**BERT-based Schema Encoding**：
```python
# Schema 序列化
schema_text = """
Table: customers
Columns: customer_id (INT, PK), customer_name (VARCHAR), email (VARCHAR)
Foreign Keys: None

Table: orders  
Columns: order_id (INT, PK), customer_id (INT, FK), total_amount (DECIMAL)
Foreign Keys: customer_id -> customers.customer_id
"""

# BERT 编码
schema_embedding = bert_encode(schema_text)
```

#### 2.2.2 Schema 序列化表示

**DIN-SQL 风格**：
```
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    email VARCHAR(255)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);
```

**DAIL-SQL 风格**（更简洁）：
```
Table customers: [customer_id, customer_name, email]
Table orders: [order_id, customer_id, order_date, total_amount]
FK: orders.customer_id -> customers.customer_id
```

### 2.3 Schema 表示的挑战

| 挑战 | 描述 | 影响 |
|------|------|------|
| **规模** | 企业级 >1000 列 | Token 限制 |
| **冗余** | 语义相似的表/列 | 召回噪声 |
| **命名** | 非自解释的列名 | 难以理解 |
| **演化** | Schema 变化 | 知识过时 |

## 3. 联合表示技术

### 3.1 Ontology-Schema 对齐

#### 3.1.1 对齐问题定义

```
给定：
- Ontology O = {E_o, R_o, A_o}  (实体、关系、属性)
- Schema S = {T_s, C_s, FK_s}  (表、列、外键)

目标：找到对齐映射 A: O ↔ S

示例：
- Ontology: Customer.hasAttribute.customer_name
- Schema: customers.customer_name
- Alignment: (Customer, customers), (customer_name, customer_name)
```

#### 3.1.2 对齐方法

| 方法 | 技术 | 准确率 | 适用场景 |
|------|------|--------|----------|
| **字符串匹配** | 编辑距离、Jaccard | ~60% | 简单场景 |
| **语义相似度** | BERT embedding | ~75% | 通用 |
| **结构匹配** | 图同构网络 | ~80% | 复杂 Schema |
| **LLM 对齐** | GPT-4 推理 | ~85% | 高资源场景 |
| **混合方法** | 多信号融合 | ~90% | 生产环境 |

#### 3.1.3 对齐挑战

```
Challenge 1: 同义词映射
- Ontology: "Customer"
- Schema: "clients", "users", "accounts"

Challenge 2: 粒度差异
- Ontology: "Address" (抽象实体)
- Schema: "street", "city", "zip_code" (具体字段)

Challenge 3: 隐含关系
- Ontology: Customer "purchases" Product
- Schema: 需要 JOIN customers -> orders -> order_items -> products
```

### 3.2 联合嵌入方法

#### 3.2.1 统一表示空间

```
Joint Embedding Space
        │
        │  ● Customer (Ontology)
        │   ● customers (Schema)
        │    ● client (User Language)
        │
        └──────────────────────
              Semantic Space
```

**目标**：使得语义等价的元素在向量空间中距离相近。

#### 3.2.2 多模态融合

```python
class JointRepresentation:
    def __init__(self, ontology, schema):
        self.ontology = ontology
        self.schema = schema
    
    def encode(self):
        # 1. 语义编码
        sem_emb = self.semantic_encoder(ontology + schema)
        
        # 2. 结构编码
        struct_emb = self.structural_encoder(schema_graph)
        
        # 3. 规则编码
        rule_emb = self.rule_encoder(business_rules)
        
        # 4. 融合
        joint_emb = self.fusion(sem_emb, struct_emb, rule_emb)
        return joint_emb
```

#### 3.2.3 表示方法对比

| 方法 | 维度 | 优势 | 劣势 |
|------|------|------|------|
| **拼接** | [ont_emb; schema_emb] | 简单 | 信息冗余 |
| **注意力** | Attention(ont_emb, schema_emb) | 自适应融合 | 计算成本 |
| **图融合** | Unified Graph + GNN | 结构保留 | 复杂度高 |
| **对比学习** | Contrastive Loss | 对齐增强 | 需要负样本 |

## 4. 召回技术

### 4.1 召回流程

```
User Query
    │
    ▼
┌─────────────────────┐
│  Query Understanding │
│  - Intent Detection  │
│  - Entity Extraction │
│  - Concept Linking   │
└─────────────────────┘
    │
    ▼
┌─────────────────────┐
│  Multi-Source Recall│
│  - Semantic Recall   │
│  - Structural Recall │
│  - Rule-based Recall │
└─────────────────────┘
    │
    ▼
┌─────────────────────┐
│  Fusion & Ranking   │
│  - Score Aggregation │
│  - Re-ranking        │
└─────────────────────┘
    │
    ▼
  Top-K Results
(Tables, Columns, Knowledge)
```

### 4.2 语义召回

#### 4.2.1 向量检索

```python
def semantic_recall(query, schema_elements, k=10):
    # 1. 编码查询
    query_emb = encoder.encode(query)
    
    # 2. 计算相似度
    scores = cosine_similarity(query_emb, schema_embeddings)
    
    # 3. Top-K
    top_k_indices = np.argsort(scores)[-k:]
    return [schema_elements[i] for i in top_k_indices]
```

#### 4.2.2 稀疏检索

```python
def sparse_recall(query, schema_elements, k=10):
    # BM25
    scores = bm25(query, schema_descriptions)
    top_k = np.argsort(scores)[-k:]
    return [schema_elements[i] for i in top_k]
```

### 4.3 结构召回

#### 4.3.1 外键传播

```
Query mentions: "customer orders"
    │
    ▼
Recall: customers, orders
    │
    ▼ (通过 FK)
Propagate: order_items (orders → order_items)
    │
    ▼ (通过 FK)
Propagate: products (order_items → products)
```

#### 4.3.2 路径召回

```python
def path_recall(start_table, end_table, schema_graph, max_hops=3):
    paths = bfs(schema_graph, start_table, end_table, max_hops)
    tables_in_path = set()
    for path in paths:
        tables_in_path.update(path)
    return tables_in_path
```

### 4.4 规则召回

#### 4.4.1 业务规则匹配

```python
def rule_recall(query, business_rules):
    matched_knowledge = []
    for rule in business_rules:
        if rule.matches(query):
            matched_knowledge.extend(rule.get_related_entities())
    return matched_knowledge
```

**示例规则**：
```
Rule: "高价值客户"
Definition: customer_id IN (
    SELECT customer_id FROM orders 
    GROUP BY customer_id 
    HAVING SUM(total_amount) > 10000
)
Related Tables: customers, orders
Related Columns: customer_id, total_amount
```

### 4.5 混合召回

```python
def hybrid_recall(query, k=10):
    # 1. 语义召回
    sem_results = semantic_recall(query, k=k*2)
    
    # 2. 结构召回
    struct_results = structural_recall(query, k=k*2)
    
    # 3. 规则召回
    rule_results = rule_recall(query, k=k*2)
    
    # 4. 融合
    all_results = merge(sem_results, struct_results, rule_results)
    
    # 5. 重排序
    ranked_results = rerank(query, all_results)
    
    return ranked_results[:k]
```

### 4.6 召回评估

| 指标 | 定义 | 公式 |
|------|------|------|
| **Recall@K** | 前K结果包含相关元素的比例 | #relevant_in_topK / #total_relevant |
| **Precision@K** | 前K结果的准确率 | #relevant_in_topK / K |
| **MRR** | 平均倒数排名 | mean(1/rank) |
| **NDCG** | 归一化折损累积增益 | DCG / IDCG |

## 5. SQL 生成技术

### 5.1 模板方法

```python
templates = {
    "simple_select": "SELECT {columns} FROM {table} WHERE {conditions}",
    "aggregation": "SELECT {agg_func}({column}) FROM {table} GROUP BY {group_by}",
    "join": "SELECT {columns} FROM {t1} JOIN {t2} ON {join_cond}"
}

def template_generate(intent, slots):
    template = templates[intent]
    sql = template.format(**slots)
    return sql
```

**局限性**：覆盖有限，无法处理复杂查询。

### 5.2 Seq2Seq 方法

```
Encoder: Query + Schema → Hidden States
Decoder: Hidden States → SQL Tokens
```

**代表模型**：
- IRNet
- RAT-SQL
- GraPPa

### 5.3 LLM 方法

#### 5.3.1 Zero-Shot

```python
prompt = f"""
Schema: {schema}
Query: {query}
Generate SQL:
"""
sql = llm.generate(prompt)
```

#### 5.3.2 Few-Shot

```python
prompt = f"""
Examples:
{few_shot_examples}

Schema: {schema}
Query: {query}
Generate SQL:
"""
sql = llm.generate(prompt)
```

#### 5.3.3 Chain-of-Thought (DIN-SQL)

```
Step 1: Identify relevant tables and columns
Step 2: Determine the join conditions
Step 3: Identify the select clause
Step 4: Build the WHERE clause
Step 5: Add GROUP BY / ORDER BY if needed
Step 6: Combine into final SQL
```

### 5.4 知识引导生成

```python
def knowledge_guided_generation(query, recalled_knowledge):
    prompt = f"""
## Recalled Knowledge
Tables: {recalled_knowledge.tables}
Columns: {recalled_knowledge.columns}
Business Rules: {recalled_knowledge.rules}

## Query
{query}

## Task
Generate SQL using ONLY the recalled tables and columns.
"""
    sql = llm.generate(prompt)
    return sql
```

## 6. 技术方法对比总结

### 6.1 表示方法对比

| 维度 | 传统方法 | 现代方法 | 本文方向 |
|------|----------|----------|----------|
| **Ontology** | RDF/OWL | KG Embedding | 混合表示 |
| **Schema** | DDL | GNN Embedding | 语义增强 |
| **联合** | 手动对齐 | 自动对齐 | 联合嵌入 |

### 6.2 召回方法对比

| 方法 | 信号 | 优势 | 劣势 |
|------|------|------|------|
| **语义召回** | 文本相似度 | 简单高效 | 忽略结构 |
| **结构召回** | 图拓扑 | 保留关系 | 忽略语义 |
| **规则召回** | 业务规则 | 领域知识 | 规则维护 |
| **混合召回** | 多信号 | 全面 | 复杂度高 |

### 6.3 SQL 生成方法对比

| 方法 | 复杂度 | 泛化性 | 准确率 |
|------|--------|--------|--------|
| **模板** | O(1) | 低 | 高（受限） |
| **Seq2Seq** | O(n) | 中 | 中 |
| **LLM Zero-Shot** | O(n) | 高 | 中 |
| **LLM Few-Shot** | O(n) | 高 | 高 |
| **知识引导** | O(n+k) | 高 | 最高 |

## 7. 技术方法总结

### 7.1 核心洞察

1. **表示**：需要联合表示框架，而非独立编码
2. **召回**：多信号融合优于单一信号
3. **生成**：知识约束可显著提升准确率

### 7.2 本文技术路线

```
P5 方案设计方向
├── 表示层：Ontology-Schema 联合嵌入
├── 召回层：语义 + 结构 + 规则混合召回
├── 生成层：知识引导的 LLM SQL 生成
└── 评估层：端到端 + 组件级评估
```

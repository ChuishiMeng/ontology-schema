## 🏗️ T5.2 Ontology 构建方案

### 2.1 标注粒度设计

#### 三级标注体系

```
┌─────────────────────────────────────────────────────────────┐
│                   Ontology 三级标注体系                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  L1: 表级 Ontology (Table-Level)                            │
│  ├── 业务实体定义: 会员、商品、订单                          │
│  ├── 实体属性: 会员属性、商品属性                            │
│  └── 映射: Table → Business Entity                          │
│                                                              │
│  L2: 列级 Ontology (Column-Level)                           │
│  ├── 属性映射: 列 → 实体属性                                 │
│  ├── 类型约束: 数值型、文本型、日期型                        │
│  └── 业务含义: 注册日期、会员等级                            │
│                                                              │
│  L3: 值级 Ontology (Value-Level)                            │
│  ├── 术语定义: 金牌会员 = tier='gold'                       │
│  ├── 范围定义: 大额订单 = amount > 10000                    │
│  └── 枚举值: 分类层级、状态码                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 标注流程设计

#### 半自动标注流程

```
Step 1: Schema 分析 (自动)
├── 输入: 数据库 Schema
├── 处理: 提取表、列、外键关系
└── 输出: Schema 结构图

Step 2: 候选 Ontology 生成 (自动 + LLM)
├── 输入: Schema 结构 + 示例数据
├── 处理: LLM 生成候选业务实体和术语
└── 输出: 候选 Ontology 草案

Step 3: 人工校验 (人工)
├── 输入: 候选 Ontology 草案
├── 处理: 领域专家校验、修正
└── 输出: 校验后的 Ontology

Step 4: 问题匹配验证 (自动)
├── 输入: Ontology + 数据集问题
├── 处理: 匹配问题中的业务术语
└── 输出: 匹配率统计

Step 5: 迭代优化 (人工 + 自动)
├── 输入: 匹配率报告
├── 处理: 补充缺失 Ontology 项
└── 输出: 最终 Ontology
```

## 📈 T5.3 评估指标定义

### 3.1 召回指标 (Schema Linking)

#### 核心指标

| 指标 | 定义 | 公式 | 优先级 |
|------|------|------|--------|
| **Precision** | 召回的 Schema 项中正确的比例 | TP / (TP + FP) | ⭐⭐⭐⭐ |
| **Recall** | 必需的 Schema 项中被召回的比例 | TP / (TP + FN) | ⭐⭐⭐⭐⭐ |
| **F1** | Precision 和 Recall 的调和平均 | 2PR / (P + R) | ⭐⭐⭐⭐ |

#### 细分指标

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| **Table Precision/Recall** | 表级召回性能 | 按表计算 TP/FP/FN |
| **Column Precision/Recall** | 列级召回性能 | 按列计算 TP/FP/FN |
| **Strict Recall** | 严格召回（RSL-SQL 定义） | 所有必需项都召回 |
| **Soft Recall** | 宽松召回（允许部分遗漏） | 主要必需项召回 |

#### Ontology 召回指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **Ontology Precision** | 召回的 Ontology 项中正确的比例 | 验证业务术语映射 |
| **Ontology Recall** | 问题提及的 Ontology 项被召回的比例 | 关键指标 |
| **Term Coverage** | 业务术语覆盖率 | 问题中术语被正确映射的比例 |

### 3.2 执行准确率指标 (End-to-End)

#### 标准指标

| 指标 | 定义 | 说明 | 优先级 |
|------|------|------|--------|
| **EX (Execution Accuracy)** | 执行结果与参考结果一致 | 主流指标，允许 SQL 语法不同 | ⭐⭐⭐⭐⭐ |
| **EM (Exact Match)** | SQL 字符串完全匹配 | 严格指标，可能有假阴性 | ⭐⭐⭐⭐ |
| **VES (Valid Efficiency Score)** | 有效执行效率评分 | 考虑执行时间和正确性 | ⭐⭐⭐ |

#### EX 计算细节

```
EX = 1 if Execute(Predicted SQL) == Execute(Ground Truth SQL)
EX = 0 otherwise

注意:
- 结果集作为多重集比较（Multiset）
- 忽略列顺序差异
- 数值在误差范围内视为相等
```

#### 难度分层指标

| 难度级别 | 描述 | 示例 |
|----------|------|------|
| **Easy** | 单表查询，无 JOIN | SELECT * FROM table WHERE condition |
| **Medium** | 多表 JOIN，简单聚合 | SELECT ... FROM t1 JOIN t2 GROUP BY ... |
| **Hard** | 嵌套查询，复杂聚合 | SELECT ... FROM (SELECT ...) GROUP BY ... HAVING ... |
| **Extra Hard** | 多嵌套，窗口函数 | WITH cte AS (...) SELECT ... OVER (...) |

#### Ontology 增强指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **Ontology-EX** | 使用 Ontology 后的 EX 提升 | ΔEX = EX_with_Ontology - EX_without_Ontology |
| **Term Resolution Rate** | 业务术语解析成功率 | 术语被正确映射到 Schema 的比例 |
| **Ontology Coverage** | Ontology 覆盖率 | 问题中业务概念被 Ontology 覆盖的比例 |

### 3.3 效率指标

#### 延迟指标

| 指标 | 定义 | CHESS 参考 | 本研究目标 |
|------|------|------------|-----------|
| **Total Latency** | 端到端总延迟（问题 → SQL） | ~10s | < 10s |
| **Schema Linking Latency** | Schema Linking 阶段延迟 | ~3s | < 5s |
| **Ontology Retrieval Latency** | Ontology 召回延迟 | — | < 500ms |
| **LLM Call Latency** | 单次 LLM 调用延迟 | ~3s | < 3s |

#### Token 消耗指标

| 指标 | 定义 | CHESS/CodeS 参考 | 本研究目标 |
|------|------|------------------|-----------|
| **Input Tokens** | 单次请求输入 Token 数 | ~8K (CHESS) | < 6K |
| **Output Tokens** | 单次请求输出 Token 数 | ~0.5K | < 0.5K |
| **Total Tokens/Query** | 每查询总 Token 消耗 | ~1600 (DAIL) | < 3K |
| **Cost per Query** | 每查询成本（$） | ~$0.02 | < $0.03 |

#### LLM 调用次数对比

| 方法 | LLM 调用次数 | 说明 |
|------|-------------|------|
| **CHESS** | ~6 次（低配） | 四Agent + 修订循环 |
| **DIN-SQL** | 4+ 次 | 四步分解 + Self-Correction |
| **DAIL-SQL** | 1 次 | Few-shot prompt |
| **本研究** | **2-4 次** | 联合召回 + SQL生成 + 验证（P4诚实分析）|

#### 资源效率指标

| 指标 | 定义 | 说明 |
|------|------|------|
| **LLM Call Count** | 每查询 LLM 调用次数 | 目标：≤ 4 次 |
| **Cache Hit Rate** | Ontology 缓存命中率 | 目标：> 80% |
| **Memory Usage** | 运行时内存占用 | 目标：< 4GB |

### C. 评估脚本设计

#### Schema Linking 评估

```python
def evaluate_schema_linking(predicted, ground_truth):
    """
    评估 Schema Linking 性能
    
    Args:
        predicted: 召回的 Schema 项 {tables, columns}
        ground_truth: 必需的 Schema 项 {tables, columns}
    
    Returns:
        precision, recall, f1
    """
    tp_tables = len(predicted['tables'] & ground_truth['tables'])
    fp_tables = len(predicted['tables'] - ground_truth['tables'])
    fn_tables = len(ground_truth['tables'] - predicted['tables'])
    
    tp_columns = len(predicted['columns'] & ground_truth['columns'])
    fp_columns = len(predicted['columns'] - ground_truth['columns'])
    fn_columns = len(ground_truth['columns'] - predicted['columns'])
    
    precision = (tp_tables + tp_columns) / (tp_tables + tp_columns + fp_tables + fp_columns)
    recall = (tp_tables + tp_columns) / (tp_tables + tp_columns + fn_tables + fn_columns)
    f1 = 2 * precision * recall / (precision + recall)
    
    return precision, recall, f1
```

#### Ontology 召回评估

```python
def evaluate_ontology_retrieval(predicted_ontology, ground_truth_ontology):
    """
    评估 Ontology 召回性能
    
    Args:
        predicted_ontology: 召回的 Ontology 项
        ground_truth_ontology: 问题中提及的 Ontology 项
    
    Returns:
        ontology_precision, ontology_recall, term_coverage
    """
    # 计算 Ontology Precision/Recall
    tp = len(predicted_ontology & ground_truth_ontology)
    fp = len(predicted_ontology - ground_truth_ontology)
    fn = len(ground_truth_ontology - predicted_ontology)
    
    precision = tp / (tp + fp) if (tp + fp) > 0 else 0
    recall = tp / (tp + fn) if (tp + fn) > 0 else 0
    
    return precision, recall
```

## 📋 v1.1 补充（2026-04-14）

### 基于 P2 文献补充的关键修订

#### 1. BIRD Knowledge Evidence 机制（新增）

BIRD 提供了 **knowledge evidence K** 作为外部知识辅助：

$$Y = f(Q, D, K | \theta)$$

**Knowledge Evidence 示例**：
- "POPLATEK TYDNE stands for weekly issuance"
- "The condition of loans is that the type of account should be 'OWNER'"

**本研究与 BIRD Knowledge Evidence 的区别**：

| 维度 | BIRD Knowledge Evidence | 本研究 Ontology |
|------|------------------------|----------------|
| 形式 | 非结构化文本片段 | 结构化图谱 |
| 推理 | 无推理机制 | 元路径约束推理 |
| 召回 | 无筛选机制 | 最小完备集提取 |
| 构建 | 数据集提供（已标注） | LLM 半自动构建 |
| 覆盖 | 仅部分问题有 evidence | 全覆盖（Ontology 层） |

#### 2. Sequeda Benchmark 对比（新增）

Sequeda et al. (2023) 提供了 KG 对 Text-to-SQL 影响的评估基准：

| 数据集 | 领域 | 表数 | 问题数 | SQL 准确率 | KG 准确率 |
|--------|------|------|--------|-----------|----------|
| Sequeda Benchmark | 保险 | 13 | 43 | 16.7% | 54.2% |

**关键发现**：
- >5 表时 SQL 准确率降至 **0%**（支持最小完备集必要性）
- KG 提升 3x 准确率（可作为本研究 Motivation）

#### 3. Ontology 构建成本对比（新增）

| 方法 | 构建方式 | 成本 | 本研究目标 |
|------|----------|------|-----------|
| ATHENA (VLDB 2016) | 手工 OWL2 | 数周/数据库 | — |
| Sequeda (2023) | 手工 OWL+R2RML | 数周/数据库 | — |
| OPTIQUE | 半自动 | 中等 | — |
| **本研究** | LLM 半自动 + 人工校验 | **≤2 小时/数据库** | ✅ |
